## [Unreleased]

### Added
- **Longhorn distributed block storage is live (v2.0's first phase)** — chart `longhorn/longhorn` 1.12.0 pinned, installed into the upstream-conventional `longhorn-system` namespace (a custom namespace was verified workable via `helm template` — zero hardcoded refs — then rejected for the permanent doc/tooling friction, ADR-0032). The disks were already waiting: ADR-0021's per-worker ext4 data disks mounted at `/var/lib/longhorn` needed zero changes. Data-replica policy is two StorageClass tiers — `longhorn` (1 copy, dev) and `longhorn-replicated` (2 copies, hard-anti-affine across the workers, prod) — because a PVC has no replica field of its own and pod replicas ≠ data replicas (ADR-0030); neither class is the cluster default, dodging k3s's local-path manifest-reapply gotcha (ADR-0031). Longhorn's own control path is trimmed to homelab scale (CSI sidecars and UI at 1 replica — leader-election HA nobody's paying for here). The auth-less UI is Tailscale-only at `https://longhorn.in.neovara.uk` via the standard internal IngressRoute pattern (ADR-0033). New `longhorn`-tagged Ansible tasks in `k3s_agent` ensure `iscsid` running + `nfs-common` on workers. Smoke-tested end to end: PVC on the prod tier → busybox pod wrote and read back data → UI showed the volume Healthy with one replica per worker → pod deleted and re-created, data survived rescheduling; PVC deletion confirmed as the actual data-deletion step (`k8s/longhorn/`).
- **Internal access path is live (Phase 12 Part C)** — `https://whoami.in.neovara.uk` works from tailnet devices only, with a real Let's Encrypt cert, completing the cluster half of ADR-0018 and putting the Phase 10 wildcard cert into actual service. Three manifests: a third Traefik Service claimed by the Tailscale Operator (`loadBalancerClass: tailscale`, hostname pinned to `traefik-internal` — `k8s/traefik/tailscale-service.yaml`), a default `TLSStore` wiring `internal-wildcard-tls` in as Traefik's default cert so any IngressRoute's empty `tls: {}` serves it cross-namespace (`k8s/traefik/tlsstore.yaml`), and an internal IngressRoute for whoami (`k8s/example-app/ingressroute-internal.yaml`). DNS is one grey-cloud wildcard CNAME `*.in.neovara.uk → traefik-internal.<tailnet>.ts.net` — discovered en route that the public ts.net record is never published (NXDOMAIN); resolution works because Tailscale's quad-100 resolver patches the CNAME's A record from its own live state, so resolve==reach and rebuilds self-heal (ADR-0029). Verified end to end from a tailnet Mac (TLS `CN=*.in.neovara.uk`, HTTP/2 200) plus the negative test: off-tailnet, the name doesn't resolve at all.
- Terraform (`terraform/`) provisioning `k3s-server-1` (control plane) and `k3s-worker-1`/`k3s-worker-2` (workers) via the `bpg/proxmox` provider, cloned from a cloud-init template. Sizing takes nearly the whole host under a no-RAM-overcommit rule (ADR-0020); each worker carries a dedicated 280GB thin-provisioned data disk reserved for distributed storage (ADR-0021), formatted and mounted by the `k3s_agent` role. All node specs are passed explicitly via `terraform.tfvars` (declared total: 740G of the 816G pool, ~9% headroom kept against thin-pool exhaustion).
- A `terraform-token` tag in the `proxmox_host` role — idempotent bootstrap of the scoped `terraform@pve` API token via `pveum`, granting three independently-checked ACLs (VM at `/`, storage at `/storage/<pool>`, SDN network-attach at `/sdn/zones/localnetwork/<bridge>` — ADR-0004, ADR-0023, ADR-0024) after the first live `terraform apply` surfaced each gap in turn.
- **`k3s-server-1`, `k3s-worker-1`, and `k3s-worker-2` are live** — Phase 3 complete, all three VMs cloned, booted, and reachable at their static IPs via cloud-init.
- **The k3s cluster itself is live** — Phases 4-6 and 7 complete. All three nodes `Ready`, `k3s-server-1` confirmed running real embedded etcd (`control-plane,etcd` role, manual snapshot save/list verified), control-plane taint confirmed, worker data disks confirmed formatted and mounted at `/var/lib/longhorn` (label `k3s-data`).
- **Helm installed (Phase 8), Tailscale Kubernetes Operator live (Phase 12 Part B)** — `kubectl` now works from any device on the tailnet (confirmed from a second physical machine), through the Operator's API server proxy, with zero exposed ports on the LAN or internet. Required a 3-scope OAuth client (Devices Core + Auth Keys + Services — ADR-0025) and an explicit `ClusterRoleBinding` granting the Tailscale-authenticated identity `cluster-admin` (ADR-0026, `k8s/tailscale/api-server-rbac.yaml`).
- **Cloudflare Tunnel is live (Phase 11)** — remotely-managed `cloudflared` connector (token from a Secret), egress locked by NetworkPolicy to exactly Traefik/DNS/Cloudflare-edge. Public path verified end-to-end (`https://whoami.neovara.uk` reaches Traefik with the same 404 as an in-cluster hit; nothing inbound opened on the router). Along the way: fixed the tunnel service target `:8000`→`:80` (Service port, not container port), added TCP 7844 to the netpol (http2 fallback), fixed a `namespaceSelector` `matchLabels` schema bug, and — hitting Cloudflare's free Universal SSL one-level-wildcard limit — switched public naming to specific first-level names during migration (ADR-0028, `k8s/cloudflared/`).
- **v1.0 platform validated — end-to-end verification complete (Phase 13).** Public path: `https://whoami.neovara.uk` → 200 with header-echo body, load-balancing confirmed across both whoami replicas; private path: `kubectl` over Tailscale confirmed working independently. The tunnel's public-hostname config silently failed to sync to cloudflared (dashboard edit never propagated — `kubectl logs -n cloudflare | grep 'Updated to new configuration'` proved the ingress JSON still held the stale `k8s.neovara.uk` entry); deleting and re-creating the hostname from scratch fixed it. Cleaned up the throwaway whoami app. The platform is now real, end to end.
- **cert-manager is live (Phase 10)** — installed via the OCI chart (`oci://quay.io/jetstack/charts/cert-manager`, `crds.enabled=true`) after confirming that's now upstream's recommended method over the legacy HTTP-repo + separate-CRD-manifest pattern. Staging + prod Let's Encrypt `ClusterIssuer`s (DNS-01 via Cloudflare) both Ready; issued a real browser-trusted wildcard cert for `*.in.neovara.uk` (staging-tested first, then flipped to prod) into `internal-wildcard-tls` in the `traefik` namespace. Removed a stale `email` field from the Cloudflare solver blocks (applies to API-Key auth, not the API-Token auth we use) and confirmed the ClusterIssuer secret-namespace rule against live docs (`k8s/cert-manager/`).
- **Traefik is live (Phase 9)** — installed `ClusterIP`-only via Helm, IngressRoute CRDs applied separately via server-side apply. Caught the chart's values schema had drifted since the guide was written (`ports.websecure.tls` → `ports.websecure.http.tls`, `service.type` → `service.spec.type` — the latter silently defaulting to `LoadBalancer` with no error); fixed `k8s/traefik/values.yaml` and pinned the chart version going forward (ADR-0027).
- Ansible roles `common`, `k3s_server`, `k3s_agent` — OS prerequisites and k3s bootstrap/join, driven by `ansible/site.yml`.
- Ansible role `proxmox_host` (`ansible/proxmox-host.yml`) — codifies Phase 1's remaining housekeeping (disable enterprise repo, enable no-subscription repo, `apt full-upgrade`) under the `repos` tag, and joining the Proxmox host to the tailnet under the `tailscale` tag, non-interactively via a runtime-supplied auth key.
- Kubernetes manifests for Traefik (ClusterIP-only ingress), cert-manager + Cloudflare DNS-01 ClusterIssuers, `cloudflared` (Cloudflare Tunnel) with a locked-down NetworkPolicy, the Tailscale Kubernetes Operator, and a `whoami` example app for end-to-end validation.
- `GUIDE.md` — full phase-by-phase build guide with the reasoning behind every architectural decision.
- `claude.md` — durable context and decisions for AI-assisted work in this repo.

### Removed
- SOPS + age secrets scaffolding (`.sops.yaml`, `secrets/`, the worked-example encrypted Secret manifest). Deferred by design for v1 — Kubernetes Secrets are created imperatively via `kubectl create secret` and are not committed to Git in any form for now. See `claude.md` and `ROADMAP.md`.
