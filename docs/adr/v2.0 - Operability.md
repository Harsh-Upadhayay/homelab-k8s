# v2.0 — Operability (ADR-0030 … ADR-0041)

> Back to [[Homelab Learning Map]] · Previous [[v0.4 - Public and Private Access]] · Index [[README|ADR index]]

The first Operability phase cashes in a decision made back in Foundation: ADR-0021 shaped every worker with a dedicated data disk mounted at `/var/lib/longhorn`, and Longhorn v1.12.0 now runs on exactly that layout, unchanged. The decisions this phase produced are all about *policy on top of the machinery*: data-replica counts expressed as StorageClass tiers rather than per-app knobs (ADR-0030), no new cluster default StorageClass (ADR-0031), the upstream-conventional namespace after actually verifying the alternative worked (ADR-0032), and an internal-only front door for a UI that ships with no authentication at all (ADR-0033).

The second Operability phase, monitoring (kube-prometheus-stack + Loki/Alloy), reuses that same storage-tier machinery (ADR-0034) and extends the internal-front-door pattern to two more dashboards — but with a genuine new wrinkle on Grafana specifically, which had a working login to deliberately remove rather than an absence to accept (ADR-0036). The rest of the phase's decisions are about making the install actually reproducible from a clean environment: deterministic Helm naming instead of post-install `kubectl get svc` discovery (ADR-0035), datasources wired declaratively instead of via a manual UI step (ADR-0037), and a log-collection shape chosen against an explicit resource criterion rather than defaulting to the chart's own (ADR-0038).

The third Operability chunk closed out the metrics-scraping gaps that were explicitly deferred out of the monitoring install (GitHub issue #25) — Longhorn's and Traefik's own bundled ServiceMonitors, and k3s's control-plane components — plus imported real dashboards for the first two rather than leaving them scraping-but-invisible in Grafana. It also surfaced and corrected a real error from the previous phase: Prometheus's ServiceMonitor discovery was never actually cluster-wide by default, despite an earlier ADR-era assumption that it was (ADR-0039) — corrected in place here rather than left standing, since it was a factual error about chart behavior, not a considered decision being reversed. The control-plane metrics fix (ADR-0040) is this project's first change to the k3s server process itself since Phase 4–6, not just to something layered on top of it. ADR-0041 records the dashboard-import mechanism chosen for both fixes.

## ADR-0034 — Storage tier for the monitoring stack: `longhorn` (1-copy, dev tier)
**Status:** Accepted

**Context:** Prometheus, Alertmanager, Loki, and Grafana all need real PVCs — upstream ships none of them with persistence or resource requests by default (confirmed directly against each chart's own `values.yaml`: an unset `storageSpec`/`storage`/`persistence` produces an emptyDir TSDB and zero Grafana persistence). ADR-0031 already established that no StorageClass is the cluster default, so every PVC must name one explicitly, and ADR-0030 already established two tiers: `longhorn` (1 copy, dev) and `longhorn-replicated` (2 copies, prod).

**Decision:** All four stateful monitoring components use `longhorn`, not `longhorn-replicated`. Metrics and logs are inherently re-collected over time by the ongoing act of scraping/tailing live workloads — losing one worker's share of history to a node failure is a materially smaller risk than losing an application's actual data, which is what the prod tier exists to protect.

**Consequences:** A worker failure loses that worker's share of monitoring history but nothing operationally irreplaceable; Prometheus/Loki simply resume collecting once rescheduled. Sizes chosen conservatively for homelab scale: Prometheus 10Gi/15d retention, Loki 10Gi, Grafana 2Gi, Alertmanager 1Gi.

## ADR-0035 — Deterministic Helm naming instead of post-install discovery
**Status:** Accepted

**Context:** The first pass at wiring Alloy's Loki write endpoint and the Grafana IngressRoute's target Service relied on installing a chart, running `kubectl get svc` to discover whatever name Helm's default `<release>-<chart>` fullname template produced, then hand-editing the next file to match — a real obstacle to rebuilding this cluster cleanly on a new environment, flagged as such and asked about directly.

**Decision:** Grafana and Loki both pin `fullnameOverride` (`grafana`, `loki`) so their Service names are fixed values baked into the values files themselves, verified by rendering each chart locally with `helm template` and checking the resulting Service's `metadata.name` before ever touching the cluster. Prometheus's Service name is deliberately left un-pinned: the parent chart exposes only a whole-release `fullnameOverride`, confirmed (via `--set fullnameOverride=...`) to rename every other object in the release too (kube-state-metrics, the operator, every bundled dashboard/rule) — too large a blast radius for zero benefit. Its name (`kube-prometheus-stack-prometheus`) is still fully deterministic, because it derives from the release name the documented install command already fixes.

**Consequences:** Rebuilding this stack from Git on a new environment needs zero manual `kubectl get svc` steps between chart installs — every cross-reference in these manifests is a static, verified string. The Grafana/Loki-vs-Prometheus asymmetry in *how* each is pinned is deliberate: whichever mechanism achieves determinism with the smallest blast radius for that specific chart.

## ADR-0036 — Grafana and Prometheus exposed internal-only; Grafana's login removed entirely
**Status:** Accepted

**Context:** Same "reachability via the Tailscale-only IngressRoute is the sole gate" model already established for the Longhorn UI (ADR-0033) — but Grafana's starting point differs in a way worth recording precisely. Longhorn ships with *zero* authentication built in, so ADR-0033 accepted an absence there was no cheap way to fill. Grafana ships with a working login for free (chart auto-generates and stores an admin password in a Secret) — removing it is an active downgrade traded for convenience, not "accepting a gap." The blast radius also differs: Longhorn unauthenticated means destructive volume/disk operations; Grafana unauthenticated means read access to whatever ends up in application logs via the Loki datasource, plus dashboard/alerting write access under the granted role — quieter, but not nothing, especially once real application workloads (e.g. a future Kiroku deployment) start logging through this same pipeline.

**Decision:** Both Grafana and Prometheus get an internal-only IngressRoute (`grafana.in.neovara.uk`, `prometheus.in.neovara.uk`) — identical websecure/TLSStore/no-public-twin shape as every other operator dashboard on this platform. Grafana additionally disables its own login entirely: `auth.anonymous.enabled: true` with `org_role: Admin`, and `auth.disable_login_form: true` to hide the login screen rather than leave it present-but-bypassable. Prometheus needed no equivalent decision — it ships with no auth in this values file at all, so the same reachability-is-the-gate model applies to it automatically rather than as a deliberate removal.

**Consequences:** Anyone reaching `grafana.in.neovara.uk` over the tailnet gets full Grafana admin, no credential needed — accepted deliberately, on the same single-operator-tailnet premise as ADR-0033 (and the same fragility: if the tailnet is ever shared with other users, this decision needs revisiting, same as ADR-0033's own caveat). The generated admin Secret (`grafana`, key `admin-password`) still exists but is never used.

## ADR-0037 — One Grafana only; Loki wired in as a provisioned datasource
**Status:** Accepted

**Context:** An early assumption was that the Loki chart's own bundled dependencies needed a Grafana toggle disabled to avoid a second Grafana instance — checked directly against `grafana/loki`'s `Chart.yaml` and found to be wrong: its dependencies are `minio`/`grafana-agent-operator`/`rollout-operator`, no Grafana subchart at all (that bundling exists only in the older, legacy `loki-stack` chart). Separately, the original plan called for adding Loki as a Grafana datasource by hand through the UI after both charts were installed — inconsistent with wanting the whole stack to rebuild cleanly from Git with no manual steps.

**Decision:** Exactly one Grafana runs on this platform (kube-prometheus-stack's). Loki is wired to it via `grafana.additionalDataSources` in the kube-prometheus-stack values file — the same file-based provisioning mechanism the chart already uses to wire up its own Prometheus datasource, just extended with one more entry, instead of a UI click-through.

**Consequences:** A fresh install ends with both datasources present with zero post-install configuration, verified live via Grafana's own `/api/datasources` endpoint. The mechanism is generic — adding a third datasource later (e.g. Tempo, if tracing is ever added) is one more list entry, not a new pattern.

## ADR-0038 — Alloy runs as a single Deployment (API-based log collection), not the chart's default DaemonSet
**Status:** Accepted

**Context:** The chart's default shape is a DaemonSet — one Alloy pod per node, each independently discovering pods via the Kubernetes API. On this 3-node cluster that meant three collectors each doing the *entire* cluster's discovery with no partitioning between them, tripling log ingestion — caught by reasoning through the config before ever installing, not by observing duplication live. Separately, the DaemonSet as configured had no toleration for the control-plane taint, so it would silently never run on `k3s-server-1` at all.

**Decision:** Run Alloy as a single-replica Deployment, keeping the API-based discovery mechanism (`loki.source.kubernetes`, reading logs through the apiserver the same way `kubectl logs` does) rather than switching to per-node file tailing. Justified against an explicit resource criterion, not intuition: split only if a per-node pod's resource need would actually drop to roughly a third of a single pod's. It doesn't — Alloy's cost is dominated by a fixed per-process baseline (the Go runtime, its own API watch), not by this cluster's low log throughput, so three per-node pods would each cost close to a full baseline (~3x the aggregate for the same work), not a third each.

**Consequences:** One collector, no duplication, no wasted baseline overhead. It also covers the tainted control-plane node's pod logs for free — reading via the apiserver needs no presence on that node at all, unlike the file-tailing alternative which would have needed an explicit toleration, a hostPath mount, and a downward-API node-name filter. Revisit only if a genuinely log-heavy workload arrives, at which point per-node throughput would dominate the cost equation and a DaemonSet with file tailing would start paying for itself.

## ADR-0039 — Prometheus discovers ServiceMonitors/PodMonitors cluster-wide, corrected from a scoped default
**Status:** Accepted

**Context:** Issue #25 was opened on the assumption that `kube-prometheus-stack`'s Prometheus already had empty `serviceMonitorSelector`/`serviceMonitorNamespaceSelector` — "confirmed against the chart's own values," per the issue text at the time. That was wrong. Live, `kubectl get prometheus kube-prometheus-stack-prometheus -o jsonpath='{.spec.serviceMonitorSelector}'` returned `{matchLabels: {release: kube-prometheus-stack}}`, not `{}`. The chart's `serviceMonitorSelectorNilUsesHelmValues` flag (and its Pod/Rule/Probe-monitor equivalents) defaults to `true`, which auto-generates that release-scoped selector unless explicitly overridden. Consequence, confirmed live: Longhorn's and Traefik's own bundled ServiceMonitors — correctly created, correctly labeled, matching correctly-labeled Services with real endpoints behind them — never appeared in Prometheus's target list or even its generated scrape config, because neither carries the `release: kube-prometheus-stack` label (a label that only makes sense on the chart's *own* bundled ServiceMonitors).

**Decision:** Set `prometheus.prometheusSpec.serviceMonitorSelectorNilUsesHelmValues: false`, plus the matching `podMonitorSelectorNilUsesHelmValues`/`probeSelectorNilUsesHelmValues`/`ruleSelectorNilUsesHelmValues`, in `values-kube-prometheus-stack.yaml`. This is what actually produces empty (`{}`) selectors, confirmed via `helm template` before applying and via the live `Prometheus` object's spec afterward — genuine cluster-wide discovery of any ServiceMonitor/PodMonitor from any chart, in any namespace, with no per-install label required.

**Consequences:** The premise the whole monitoring phase was built on (ADR-0034–0038: "any future app just needs a ServiceMonitor next to it, Prometheus already watches everything") is now actually true, rather than true-in-the-values-file-comments-only. This is treated as a correction, not a superseding decision — no prior ADR asserted the wrong selector behavior as a considered choice, so there's nothing to mark Superseded; the wrong claim in the original monitoring-phase Concepts writeup was corrected in place instead (`docs/concepts/Kubernetes Concepts.md`).

## ADR-0040 — k3s control-plane component metrics exposed via k3s server args, scraped through hand-authored Endpoints
**Status:** Accepted

**Context:** k3s runs kube-scheduler, kube-controller-manager, and etcd inside its own single server binary rather than as separately-labeled pods (the way kubeadm does). Two independent consequences follow, both root-caused live before any fix: (1) their metrics listeners bind to `127.0.0.1` only by k3s default, unreachable from any other pod regardless of Service/ServiceMonitor config; (2) even once reachable, the chart's auto-generated Services for these three components select on pod labels that structurally can never match anything on k3s, so their `Endpoints` objects stay permanently empty (see [[Kubernetes Concepts]]).

**Decision:** Two changes, one per side of the problem. On the k3s side (`ansible/roles/k3s_server/templates/config.yaml.j2`): `kube-controller-manager-arg`/`kube-scheduler-arg: bind-address=0.0.0.0` open the existing metrics listeners beyond loopback, and `etcd-expose-metrics: true` does the same for etcd's separate, unauthenticated metrics-only port — none of this changes how these components serve traffic otherwise, only where their existing `/metrics` endpoint is reachable from. On the Helm side (`values-kube-prometheus-stack.yaml`): `kubeControllerManager`/`kubeScheduler`/`kubeEtcd.endpoints: [192.168.1.21]` make each chart author a static `Endpoints` object pointed at the real node IP and the standard upstream ports (10257/10259/2381) instead of waiting on pod-selector discovery that will never succeed; `insecureSkipVerify: true` is set for the two HTTPS endpoints (self-signed certs from k3s's internal CA, same trust boundary as every other in-cluster TLS hop on this platform) — etcd's metrics port needs no such override, since `--etcd-expose-metrics` serves plain HTTP by design.

**Consequences:** All three components confirmed `up` in Prometheus's target list, live. This is a single-server-cluster fix — `endpoints` is a one-item list today (ADR-0008); a future HA control plane (`k3s-server-2`/`-3`, deferred per `ROADMAP.md`) would need every additional server's IP added to all three lists. One live gotcha worth remembering: the chart's Services already existed with selectors matching zero pods, so Kubernetes had already auto-created empty, non-Helm-owned `Endpoints` objects with the same names — the first `helm upgrade` attempt failed on an ownership conflict, resolved by `kubectl annotate`-ing the existing objects with Helm's release ownership annotations rather than deleting and racing the endpoint-controller's near-instant recreation.

## ADR-0041 — Third-party Grafana dashboards imported via labeled ConfigMap, not manual UI import
**Status:** Accepted

**Context:** Fixing Longhorn's and Traefik's metrics scraping (ADR-0039, ADR-0040) left both correctly instrumented but with nothing to look at in Grafana — kube-prometheus-stack's 27 bundled dashboards cover only its own mixin components, never third-party apps. Manually importing a dashboard through Grafana's UI would be the one remaining click-op left in an otherwise fully declarative stack (ADR-0035, ADR-0037).

**Decision:** Both dashboards (`k8s/traefik/grafana-dashboard.yaml`, ID 17347 — Traefik's official Kubernetes dashboard; `k8s/longhorn/grafana-dashboard.yaml`, ID 17626 — the dashboard linked directly from longhorn.io's own Prometheus/Grafana docs) ship as plain `ConfigMap`s labeled `grafana_dashboard: "1"`, matching the label kube-prometheus-stack's Grafana sidecar already polls for, cluster-wide, by chart default — no extra Grafana-side config needed, same "empty selector, no per-install work" shape as ADR-0039's fix. Each dashboard's exported JSON was downloaded directly from grafana.com and checked for embedded, unresolved datasource-variable placeholders before committing it — file-based provisioning does not run the substitution step a UI import would. Traefik's needed no edits (its `templating.list` datasource variable already defaulted to the literal string `"Prometheus"`, matching this chart's own datasource name); Longhorn's older-schema dashboard had 30 bare `"${DS_PROMETHEUS-LONGHORN}"` references with nothing to resolve them, substituted directly for the literal string `"Prometheus"` ahead of time.

**Consequences:** Both dashboards exist from a fresh `kubectl apply` with zero manual Grafana steps, verified via Grafana's own `/api/search`. The pattern generalizes to any future third-party dashboard: download the JSON, check for unresolved datasource placeholders, ConfigMap-and-label it. Neither dashboard is chart-templated or version-pinned to a specific grafana.com revision — a future update means re-downloading and re-applying by hand, an accepted manual step since it's infrequent and low-risk (worst case: a stale but still-functional dashboard).

## ADR-0030 — Data-replica policy as StorageClass tiers: `longhorn` (1 copy, dev) and `longhorn-replicated` (2 copies, prod)
**Status:** Accepted

**Context:** The instinct was to set data replication per app, "in each app's manifest, like `spec.replicas`" — but that conflates two unrelated meanings of *replica*. A Deployment's `spec.replicas` is pod count (compute redundancy); Longhorn's `numberOfReplicas` is copies of a volume's data on distinct workers (data redundancy). They're orthogonal — a single-pod Postgres still wants its data to survive a node loss — and a PVC has **no replica-count field at all**: `numberOfReplicas` is a StorageClass *parameter*. Meanwhile the chart's default of 3 copies can never be satisfied on 2 workers (hard replica anti-affinity: copies never share a node), which would leave every volume permanently degraded.

**Decision:** Express data-replica policy as two StorageClass tiers sharing one driver: the chart's `longhorn` class at `numberOfReplicas: 1` for dev workloads, plus a hand-written `longhorn-replicated` at `2` (one copy per worker) for prod (`k8s/longhorn/values.yaml`, `k8s/longhorn/storageclass-replicated.yaml`). A PVC picks its tier by `storageClassName`. Longhorn's own control-path redundancy was also trimmed to fit the cluster: the four leader-elected CSI sidecar Deployments and the UI run at 1 replica each — they're provisioning-path only, never the data path, so the worst case is a new PVC waiting seconds for a reschedule.

**Consequences:** Dev data is genuinely non-redundant — a worker VM dying loses any 1-copy volume attached to it; that's the accepted trade for the tier. Prod volumes survive a worker loss (engine re-materialises on the surviving node against its local copy). ADR-0021's caveat still stands over both tiers: one physical NVMe under both workers means even 2 copies is simulated durability, real against VM failure only. Per-app replica wishes now route through class *selection*, and a third tier later is one more small manifest, not a schema change.

## ADR-0031 — Longhorn's StorageClasses are not the cluster default
**Status:** Accepted

**Context:** The chart defaults to `persistence.defaultClass: true`, but k3s's `local-path` already holds the `is-default-class` annotation — and k3s reapplies its packaged manifests on every server restart, so stripping local-path's annotation silently reverts. Two simultaneous defaults make PVC binding depend on Kubernetes-version-specific tie-breaking rules.

**Decision:** `persistence.defaultClass: false`; every PVC states `storageClassName` explicitly (`longhorn` or `longhorn-replicated`).

**Consequences:** No fight with k3s's manifest reconciler, no two-defaults ambiguity, and each PVC carries an explicit, reviewable statement of its durability requirement — a good habit for a learning platform regardless. Cost: any chart that creates PVCs without a class name gets `local-path`; revisit when the monitoring phase (which wants persistent volumes by default) lands.

## ADR-0032 — Keep the upstream-conventional `longhorn-system` namespace
**Status:** Accepted

**Context:** A custom namespace (`longhorn-ns`) was requested and verified to work — `helm template -n longhorn-ns` renders the entire chart with zero hardcoded `longhorn-system` references, so the chart is fully namespace-relative. But every upstream doc, community example, GitHub issue, and the `longhornctl` tool *assumes* `longhorn-system`, meaning every copied command would need a namespace flag added, forever.

**Decision:** Install into `longhorn-system`. Convention wins over customization that carries permanent friction and buys nothing.

**Consequences:** Upstream docs and tooling work verbatim. The verification itself was the valuable part and is kept (in `k8s/longhorn/values.yaml`'s header): the choice is documented as *considered and rejected*, not never-made — the difference between a default followed blindly and a default followed deliberately.

## ADR-0033 — Longhorn UI exposed internal-only, via the existing Tailscale front door
**Status:** Accepted

**Context:** The Longhorn UI ships with zero authentication and can delete volumes, wipe disks, and change cluster-wide storage settings. Upstream docs suggest putting an authenticating ingress in front; this platform already has a stronger equivalent — a whole private entry path where network reachability *is* the authentication (ADR-0018, ADR-0029).

**Decision:** One IngressRoute (`k8s/longhorn/ingressroute.yaml`): `longhorn.in.neovara.uk` on the `websecure` entrypoint, TLS from the default TLSStore wildcard — the exact whoami-internal pattern. Reachable only from tailnet devices; off-tailnet the name doesn't even resolve. No public (`web`-entrypoint / Cloudflare) twin, ever.

**Consequences:** Zero new auth machinery, real TLS, and the UI is as reachable as the Traefik dashboard — i.e., only by the platform's operator. The route lives in `longhorn-system` (Traefik resolves same-namespace Services only, `allowCrossNamespace` off). If the tailnet is ever shared with other users, this ADR's premise (network position = sole operator) breaks and the UI needs real auth in front.
