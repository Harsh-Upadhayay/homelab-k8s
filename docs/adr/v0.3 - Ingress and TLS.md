# v0.3 — Ingress and TLS (ADR-0013 … ADR-0015)

> Back to [[Homelab Learning Map]] · Previous [[v0.2 - Cluster Bootstrap]] · Next [[v0.4 - Public and Private Access]] · Index [[README|ADR index]]

The ingress layer, chosen deliberately to avoid stacking too many brand-new concepts on top of everything already being learned in v0.2. Traefik stays `ClusterIP`-only rather than reaching for a LoadBalancer (ADR-0013), routing uses Traefik's own native CRD instead of the ecosystem's emerging Gateway API (ADR-0014), and cert-manager is scoped narrowly to internal traffic rather than the public path, which Cloudflare already covers (ADR-0015).

---

## ADR-0013 — Traefik Service is ClusterIP only
**Status:** Accepted

**Context:** A typical bare-metal k3s ingress setup reaches for a LoadBalancer (via MetalLB or similar) to get an externally reachable IP.

**Decision:** Keep the Traefik Service as `ClusterIP` — no LoadBalancer, no MetalLB, no Cilium LB-IPAM anywhere in this design.

**Consequences:** No LAN LoadBalancer IP is ever needed, because Cloudflare Tunnel (ADR-0016) reaches Traefik entirely inside the cluster via its Service DNS name. Simpler topology, one fewer moving part (no MetalLB to install or configure) — but Traefik has zero reachability without the tunnel or a manually port-forwarded NodePort, which is a deliberate constraint, not an oversight (`EXTERNAL-IP: <none>` is correct, not broken).

## ADR-0014 — Use Traefik's native IngressRoute CRD instead of Gateway API
**Status:** Accepted

**Context:** Gateway API is the direction the Kubernetes ingress ecosystem is heading, but adopting it means installing and learning a second CRD set on top of everything else in this build.

**Decision:** Use Traefik's own `IngressRoute` CRD, which ships with the Helm chart, for all routing.

**Consequences:** One less CRD set to learn simultaneously; per-app routing stays simple to reason about. Migrating to Gateway API later is a config change, not a rebuild — unlike the CNI choice (ADR-0011), this isn't a one-way door.

## ADR-0015 — cert-manager serves internal/Tailscale-only TLS, not the public path
**Status:** Accepted

**Context:** Cloudflare already terminates TLS at its edge for anything reached through the tunnel, so a public request arrives at the cluster already over HTTPS — cert-manager on the public path would be redundant.

**Decision:** Use cert-manager with a Cloudflare DNS-01 solver only for services that are never publicly reachable (e.g. a future Grafana or Argo CD dashboard over Tailscale), not for the public-facing path.

**Consequences:** DNS-01 (rather than HTTP-01) is required specifically because these services aren't publicly reachable — an HTTP-01 challenge could never complete against them. Public traffic gets real TLS for free from Cloudflare's edge; internal-only services get real, non-self-signed certs too, via a deliberately different mechanism suited to their reachability.
