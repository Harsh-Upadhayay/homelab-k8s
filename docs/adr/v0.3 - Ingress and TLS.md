# v0.3 — Ingress and TLS (ADR-0013 … ADR-0015, ADR-0027)

> Back to [[Homelab Learning Map]] · Previous [[v0.2 - Cluster Bootstrap]] · Next [[v0.4 - Public and Private Access]] · Index [[README|ADR index]]

The ingress layer, chosen deliberately to avoid stacking too many brand-new concepts on top of everything already being learned in v0.2. Traefik stays `ClusterIP`-only rather than reaching for a LoadBalancer (ADR-0013), routing uses Traefik's own native CRD instead of the ecosystem's emerging Gateway API (ADR-0014), and cert-manager is scoped narrowly to internal traffic rather than the public path, which Cloudflare already covers (ADR-0015). The actual Phase 9 install then surfaced a gap in the pinning discipline ADR-0002 already committed to: the Traefik chart itself wasn't pinned, and it had drifted its values schema since the guide was written (ADR-0027).

---

## ADR-0013 — Traefik Service is ClusterIP only
**Status:** Accepted

**Context:** A typical bare-metal k3s ingress setup reaches for a LoadBalancer (via MetalLB or similar) to get an externally reachable IP.

**Decision:** Keep the Traefik Service as `ClusterIP` — no LoadBalancer, no MetalLB, no Cilium LB-IPAM anywhere in this design.

**Consequences:** No LAN LoadBalancer IP is ever needed, because Cloudflare Tunnel (ADR-0016) reaches Traefik entirely inside the cluster via its Service DNS name. Simpler topology, one fewer moving part (no MetalLB to install or configure) — but Traefik has zero reachability without the tunnel or a manually port-forwarded NodePort, which is a deliberate constraint, not an oversight (`EXTERNAL-IP: <none>` is correct, not broken).

## ADR-0014 — Use Traefik's native IngressRoute CRD instead of Gateway API
**Status:** Accepted

**Context:** Gateway API is the direction the Kubernetes ingress ecosystem is heading, but adopting it means installing and learning a second CRD set on top of everything else in this build.

**Decision:** Use Traefik's own `IngressRoute` CRD, which ships with the Helm chart, for all routing.

**Consequences:** One less CRD set to learn simultaneously; per-app routing stays simple to reason about. Migrating to Gateway API later is a config change, not a rebuild — unlike the CNI choice (ADR-0011), this isn't a one-way door.

## ADR-0015 — cert-manager serves internal/Tailscale-only TLS, not the public path
**Status:** Accepted

**Context:** Cloudflare already terminates TLS at its edge for anything reached through the tunnel, so a public request arrives at the cluster already over HTTPS — cert-manager on the public path would be redundant.

**Decision:** Use cert-manager with a Cloudflare DNS-01 solver only for services that are never publicly reachable (e.g. a future Grafana or Argo CD dashboard over Tailscale), not for the public-facing path.

**Consequences:** DNS-01 (rather than HTTP-01) is required specifically because these services aren't publicly reachable — an HTTP-01 challenge could never complete against them. Public traffic gets real TLS for free from Cloudflare's edge; internal-only services get real, non-self-signed certs too, via a deliberately different mechanism suited to their reachability.

## ADR-0027 — Pin the Traefik Helm chart version

**Status:** Accepted

**Context:** ADR-0002 already committed this stack to deliberate version pinning, but Phase 9's original install command (`helm install traefik traefik/traefik --namespace traefik --values k8s/traefik/values.yaml`) carried no `--version`, unlike the Phase 10 cert-manager install which does pin. Between when that command was written and when it was actually run, the chart (now `41.0.1`, app version `v3.7.5`) had restructured its values schema twice: `ports.websecure.tls` moved to `ports.websecure.http.tls` — strict enough to fail the install outright (`additional properties 'tls' not allowed`) — and `service.type` moved to `service.spec.type`, a change the schema does *not* enforce, so the old key was silently dropped and the Service defaulted to the chart's own `LoadBalancer` default with no error at all, caught only by reading `kubectl get svc`'s `TYPE` column rather than any failure.

**Decision:** Pin `--version 41.0.1` explicitly on the Traefik install (`GUIDE.md` Phase 9), and treat diffing `helm show values <repo>/<chart>` against the repo's committed values file as a standing check before any chart install/upgrade, not just a first-time step.

**Consequences:** Re-running the install later reproduces the exact same schema deterministically — no silent drift between machines or over time. Upgrading the chart becomes a deliberate, visible version bump in this repo rather than an invisible side effect of whenever `helm install` happens to run next. The `service.type` case is the sharper lesson of the two: a schema change that's silently *accepted* rather than rejected can produce a working-but-wrong install with a clean exit code — worth specifically re-checking real output (`kubectl get svc`), not just command success, after any chart install (`k8s/traefik/values.yaml`).
