# v0.4 — Public and Private Access (ADR-0016 … ADR-0018, ADR-0025 … ADR-0026, ADR-0028 … ADR-0029)

> Back to [[Homelab Learning Map]] · Previous [[v0.3 - Ingress and TLS]] · Index [[README|ADR index]]

The two front doors, and the one security tightening that makes exposing anything at all safe. Cloudflare Tunnel gets exactly one route into the cluster, with all real routing decisions pushed down to Traefik (ADR-0016) — and because that makes `cloudflared` the single public-facing component in the whole platform, its egress is locked down hard enough that compromising the exposed app doesn't hand over the rest of the cluster (ADR-0017). Private access runs on a completely separate mechanism, split further into a host half and a cluster half (ADR-0018) — both halves are now run and confirmed, and the internal DNS scheme that puts the wildcard cert into actual service closes the milestone (ADR-0029).

---

## ADR-0016 — Cloudflare Tunnel with exactly one route to Traefik
**Status:** Accepted (hostname scheme refined by ADR-0028)

**Context:** A tunnel could be configured with one route per exposed app, or one route pointed at the ingress layer that then does its own per-app routing.

**Decision:** Point every tunnel public hostname at the ingress layer — `http://traefik.traefik.svc.cluster.local:80` — and let Traefik's `IngressRoute` objects handle all per-app `Host()` matching from there. (The hostname *scheme* — single wildcard vs. specific first-level names — is refined by ADR-0028 once the Universal SSL and legacy-collision constraints surfaced. The target is the Traefik **Service** port `:80`, not the container's `:8000`.)

**Consequences:** Adding a tenth app means adding an `IngressRoute`, never touching the tunnel configuration again. Also means the tunnel config can't distinguish apps at the Cloudflare layer — any app-specific access policy has to live in Traefik or Cloudflare Access instead, not in the tunnel's route table.

## ADR-0017 — Lock down cloudflared's egress via NetworkPolicy
**Status:** Accepted

**Context:** Cloudflare's own quickstart deploys `cloudflared` with no network restriction — by default it can resolve and reach every Service in every namespace, which makes the one public-facing component in the cluster a pivot point if the exposed app is ever compromised.

**Decision:** Apply a `NetworkPolicy` restricting `cloudflared`'s egress to exactly Traefik (port 8000), DNS, and Cloudflare's own edge.

**Consequences:** An attacker who compromises the exposed app can't pivot through the tunnel pod to reach the Traefik dashboard, a database, or any other Service in the cluster. Also confirms NetworkPolicy enforcement works on plain Flannel via k3s's bundled netpol controller — this protection doesn't require Cilium (contrast ADR-0011, where Cilium is deferred for other reasons).

## ADR-0018 — Tailscale as two separate mechanisms
**Status:** Accepted

**Context:** Private access is needed for two different targets — the Proxmox hypervisor itself (UI, node SSH) and the k3s cluster (`kubectl`, in-cluster dashboards) — which could be tempted into one conflated setup.

**Decision:** Run `tailscaled` directly on the Proxmox host for hypervisor-level access, and separately install the Tailscale Kubernetes Operator in-cluster for the API server proxy and `loadBalancerClass: tailscale` Services.

**Consequences:** Each target gets private access through the mechanism that actually fits it — the host doesn't need an in-cluster Operator, and the cluster's dashboards don't need the host's `tailscaled`. Confirmed hands-on for the host half: `pve-dell` joined the tailnet and the Proxmox UI is reachable at its tailnet IP, never publicly (see [[Platform Concepts]] — mesh VPN model). The cluster half (the Operator) is now also confirmed hands-on — see ADR-0025/ADR-0026.

## ADR-0025 — Tailscale Kubernetes Operator's OAuth client needs three scopes, not one
**Status:** Accepted

**Context:** The Operator's first real deploy crash-looped: `kubectl logs --previous` showed `"creating operator authkey: calling actor does not have enough permissions to perform this function (403)"`. The OAuth client had only been granted `Devices Core: Write` — the scope that seemed obviously relevant for "manage devices on the tailnet." Confirmed via Tailscale's own docs: the Operator actually needs three scopes together — **Devices Core** (manage its own device), **Auth Keys** (it creates auth keys at runtime, for itself and for every proxy pod it later spins up per exposed Service), and **General/Services** — all tagged `tag:k8s-operator`. Tailscale also doesn't allow editing an existing OAuth client's scopes, only creating a new one.

**Decision:** Generate a fresh OAuth client with all three scopes (Read+Write) tagged `tag:k8s-operator`, and re-run the same `helm upgrade --install` with the new credentials — same release name, so Helm updates the existing Secret in place and the pod restarts to pick it up.

**Consequences:** Same class of lesson as ADR-0023/ADR-0024 (Proxmox's `PVEVMAdmin` not covering storage or SDN privileges) — a scope that *sounds* sufficient for a stated purpose often isn't, for tools built by someone else, and the fastest way to find the true minimal set is triggering the actual 403 and reading the exact error, not guessing from the name of the first plausible-looking scope. `kubectl logs --previous` was essential here — `kubectl describe pod` alone showed *that* it was crash-looping and roughly *where* (after Kubernetes API access, before completing startup), but not *why*; the specific, actionable reason only existed in the crashed container's own stderr.

## ADR-0026 — Grant Tailscale-authenticated users cluster-admin via RBAC
**Status:** Accepted

**Context:** With the Operator running and the API server proxy live, `kubectl get nodes` from a second device (a MacBook, authenticated to the same tailnet) returned `Error from server (Forbidden): ... User "harshupadhayay906@gmail.com" cannot list resource "nodes"`. This was proof the whole chain up to that point was actually working — the proxy correctly authenticated the Mac's tailnet identity and forwarded the request impersonating that exact login (`Impersonate-User`) — but Kubernetes RBAC had never heard of that identity before, so by default it had zero permissions. Authentication (who is this) and authorization (what can they do) are strictly separate systems in Kubernetes; the proxy only solves the first one.

**Decision:** Apply a `ClusterRoleBinding` (`k8s/tailscale/api-server-rbac.yaml`) binding the exact tailnet login email, as a `User` subject, to the built-in `cluster-admin` `ClusterRole` — the same role k3s's own `system:masters` group (and therefore the original admin `kubeconfig` from Phase 4–6) is already bound to. Full `cluster-admin`, not a narrower role, deliberately: this is a single-operator personal homelab with no other tailnet users, so there's no meaningful distinction between "the owner" and "an admin" to enforce — same reasoning already applied to Longhorn's `replicas=1` and the no-HA decisions.

**Consequences:** Every device on the tailnet, authenticated as this one Tailscale login, now has full cluster access through the proxy — confirmed working end to end from a second physical machine, which is the actual multi-client requirement this phase existed to satisfy. Revisit this binding (narrow it, or add per-user/per-group grants instead) the moment a second human is ever added to this tailnet — `cluster-admin`-for-everyone stops being a reasonable simplification the instant "everyone" is more than one person.

## ADR-0028 — Public hostnames use specific first-level names during migration, not a nested wildcard
**Status:** Accepted (refines ADR-0016)

**Context:** ADR-0016 assumed a single *wildcard* tunnel route. Making that concrete surfaced two stacked constraints. (1) A pre-existing legacy homelab already owns `*.neovara.uk → old router`, so the tunnel can't reuse that exact wildcard. The obvious fix — a nested `*.k8s.neovara.uk` — then hit (2): Cloudflare's **free Universal SSL only covers the root + one wildcard level** (`neovara.uk`, `*.neovara.uk`). A two-level name like `whoami.k8s.neovara.uk` gets no edge certificate at all — its TLS handshake fails at the edge (`sslv3 alert handshake failure`) *before* the tunnel is consulted (verified: the presented edge cert's SAN is exactly `DNS:neovara.uk, DNS:*.neovara.uk`). Covering `*.k8s.neovara.uk` requires the paid Advanced Certificate Manager.

**Decision:** During the migration window, give each public app a **specific first-level** hostname (`whoami.neovara.uk`, `app.neovara.uk`), each added as its own tunnel public hostname (which auto-creates its proxied DNS record). Universal SSL's `*.neovara.uk` covers them, and because a specific record beats the legacy wildcard, each name cuts over independently while everything else keeps hitting the old router. Internal Tailscale names keep the two-level `*.in.neovara.uk` — unaffected, since they use cert-manager's Let's Encrypt cert, not Cloudflare's edge.

**Consequences:** Per public app during migration = one tunnel public-hostname (auto-DNS) + one `IngressRoute` — two small additive, reversible steps, no hand-edited DNS; the `cloudflared` Deployment never changes. The one-wildcard-route simplicity of ADR-0016 is deferred to the **end-state**: once every workload is on k8s and the legacy `*.neovara.uk → router` wildcard is retired, point the tunnel at a single `*.neovara.uk` route (first-level, so still free Universal SSL) — after which adding an app is only an `IngressRoute`, zero Cloudflare/DNS change. Paying for Advanced Certificate Manager would buy back the single-wildcard model immediately (as `*.k8s.neovara.uk`); rejected to stay free for a homelab. Corrected in passing: the tunnel service target is `:80` (the Traefik *Service* port), not `:8000` (the container port) — connecting to the ClusterIP on 8000 reaches nothing, since kube-proxy only maps the Service port 80 → pod 8000 (see [[Kubernetes Concepts]]).

## ADR-0029 — Internal DNS via wildcard CNAME to the proxy's pinned MagicDNS name
**Status:** Accepted (completes the cluster half of ADR-0018)

**Context:** With the internal front door built — a `loadBalancerClass: tailscale` Service for Traefik (`k8s/traefik/tailscale-service.yaml`, proxy device `traefik-internal`), the Phase 10 wildcard cert wired in as Traefik's default via a `TLSStore` (`k8s/traefik/tlsstore.yaml`), and a first internal `IngressRoute` (`k8s/example-app/ingressroute-internal.yaml`) — the remaining question was how `*.in.neovara.uk` should *resolve*, for tailnet devices only. Two candidate schemes: (1) a hardcoded **A record** to the proxy's Tailscale IP — rejected because Tailscale 100.x IPs are stable per device *registration*, and for operator-managed proxies deleting/recreating the Service registers a **new** device with a new IP, so the record goes **silently stale on every rebuild**; (2) a **CNAME** to the proxy's MagicDNS name, which the original plan assumed required Tailscale publishing the name in *public* ts.net DNS (documented behavior with the tailnet's HTTPS-certificates feature enabled). Empirically, the public record for `traefik-internal.egret-pence.ts.net` is **NXDOMAIN** — never published, even with HTTPS certs enabled and well past the documented ~10-minute propagation — yet on-tailnet resolution worked anyway, via a better mechanism: a tailnet device's resolver is Tailscale's quad-100 (`100.100.100.100`); the query for `whoami.in.neovara.uk` is forwarded upstream, comes back as a CNAME whose target is a MagicDNS name **quad-100 itself owns**, and quad-100 fills in the A record from its own live state. No public record needed.

**Decision:** One grey-cloud (DNS-only) wildcard record in Cloudflare — `*.in.neovara.uk CNAME traefik-internal.egret-pence.ts.net` — with the proxy's device name pinned via the `tailscale.com/hostname` annotation, and the wildcard cert served to every internal route by the default `TLSStore` (an `IngressRoute`'s `tls.secretName` only works from its *own* namespace; the store makes an empty `tls: {}` serve the wildcard from any namespace).

**Consequences:** The set of clients that can **resolve** an internal name equals the set that can **reach** its IP — off-tailnet, `*.in` names aren't just unreachable but *invisible* (the public chain dies at the unpublished ts.net hop; verified `curl: (6) Could not resolve host`, and `dig @1.1.1.1` shows the CNAME in the answer but overall NXDOMAIN — no CGNAT IP leaked in practice). More self-healing than a hardcoded A record: quad-100 always knows the proxy's *current* IP, so a cluster rebuild that mints a new device IP heals automatically — **provided** the hostname stays pinned via the annotation *and* the stale tailnet device is deleted in the admin console first (otherwise the new device comes up name-suffixed `traefik-internal-1` and the CNAME breaks). Caveat, recorded not fixed: a tailnet device that bypasses Tailscale DNS (custom resolver) cannot resolve `*.in` names — acceptable for a single operator with default clients. Verified end to end from a tailnet Mac: `https://whoami.in.neovara.uk` resolved through the CNAME to `100.79.208.52`, TLS verified against `CN=*.in.neovara.uk` (Let's Encrypt, no `-k`), HTTP/2 200.
