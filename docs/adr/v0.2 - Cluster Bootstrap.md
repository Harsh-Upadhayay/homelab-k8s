# v0.2 — Cluster Bootstrap (ADR-0008 … ADR-0012)

> Back to [[Homelab Learning Map]] · Previous [[v0.1 - Foundation]] · Next [[v0.3 - Ingress and TLS]] · Index [[README|ADR index]]

This is the actual k3s architecture — decided before any of the surrounding Ansible automation existed, and encoded directly in `ansible/roles/k3s_server/templates/config.yaml.j2`. Every flag in that template is one of these decisions made concrete: embedded etcd instead of the simpler SQLite default (ADR-0008), encryption at rest from cluster-init onward rather than retrofitted later (ADR-0009), a taint that keeps the control plane free of application noise (ADR-0010), and k3s's own bundled Traefik/ServiceLB switched off to make room for the versions this repo installs deliberately (ADR-0012). One decision here outranks the rest: staying on Flannel and deferring Cilium (ADR-0011) is the single non-additive call in the entire platform.

---

## ADR-0008 — Use embedded etcd instead of SQLite
**Status:** Accepted

**Context:** k3s defaults to a SQLite datastore for single-server setups; embedded etcd is normally reserved for multi-server HA configurations.

**Decision:** Enable embedded etcd via `cluster-init: true` even at a single server node.

**Consequences:** Real etcd snapshot/restore/inspection is available immediately, and adding `k3s-server-2`/`-3` later for HA becomes a clean join rather than a datastore migration. Slightly heavier than SQLite for a single node — accepted deliberately for the hands-on etcd learning goal and the clean path to HA it opens up.

## ADR-0009 — Enable secrets-encryption at rest
**Status:** Accepted

**Context:** By default, Kubernetes Secrets are stored as base64 in etcd — readable by anyone with etcd access, not actually encrypted.

**Decision:** Set `secrets-encryption: true` in the k3s server config.

**Consequences:** Secrets are encrypted at rest in etcd from cluster-init onward. This has to be enabled at bootstrap; turning it on retroactively on an existing cluster is far more involved (requires re-encrypting every existing secret), so doing it now avoids that cost entirely.

## ADR-0010 — Taint the control-plane node against application workloads
**Status:** Accepted

**Context:** A single-server k3s cluster will schedule application pods onto the control-plane node by default, competing with etcd and the API server for resources.

**Decision:** Apply `node-role.kubernetes.io/control-plane:NoSchedule` to `k3s-server-1` via `node-taint` in its config.

**Consequences:** Application pods physically cannot land on the control plane, protecting etcd's latency sensitivity from noisy-neighbor workloads. Requires at least one worker node for anything to actually run — already satisfied since `k3s-worker-1` is provisioned from the start (see [[v0.1 - Foundation]]).

## ADR-0011 — Flannel + kube-proxy on defaults; defer Cilium
**Status:** Accepted

**Context:** Cilium is a more capable CNI — eBPF dataplane, richer NetworkPolicy, built-in observability — but adds real complexity on top of everything else already being learned in this build (etcd, ingress, tunnels, mesh VPN).

**Decision:** Leave the CNI on k3s's defaults (Flannel + kube-proxy) for v1. Treat a Cilium migration as its own dedicated project later, not a v1 concern.

**Consequences:** This is the one non-additive decision in the whole platform. Flannel → Cilium requires a full cluster rebuild, not a live migration — pod networking can't be swapped underneath a running cluster. Every other deferred item (GitOps, monitoring, secrets, backups, HA) is purely additive; this one alone means "don't casually suggest switching CNIs," and a future Cilium adoption doubles as a genuine rebuild-from-Git/DR drill rather than a routine upgrade.

## ADR-0012 — Disable k3s's bundled Traefik and ServiceLB
**Status:** Accepted

**Context:** k3s ships with a default Traefik install and a `servicelb` LoadBalancer implementation, both of which would collide with the versions this repo installs deliberately via Helm in Phase 9.

**Decision:** Set `disable: [traefik, servicelb]` in the k3s server config.

**Consequences:** A clean slate for the Helm-installed Traefik, with no fighting over the `traefik` namespace or Service ports. Means nothing is exposed at all until Phase 9 actually runs — acceptable, since that's the intended build order.
