# v0.1 — Foundation (ADR-0001 … ADR-0007, ADR-0019)

> Back to [[Homelab Learning Map]] · Next [[v0.2 - Cluster Bootstrap]] · Index [[README|ADR index]]

Foundation covers the Proxmox host itself and the tooling boundary that provisions everything above it — the cluster doesn't exist yet at this stage, so these decisions are about repo hygiene and tool boundaries rather than cluster architecture. Provisioning and configuration are kept as two separate tools with two separate mental models (ADR-0001), versions are pinned everywhere on principle (ADR-0002), and secrets management is explicitly pushed out of v1's scope rather than half-built (ADR-0003). The Proxmox-host automation itself went through a real reversal worth keeping visible: it started as two Ansible roles, got merged into one once tags proved sufficient (ADR-0005, superseded by ADR-0006) — and a portability bug in the inventory (a personal SSH alias standing in for a real address) got caught and fixed (ADR-0007). ADR-0019 landed later (numbered after the v0.2–v0.4 batch, per append-only numbering) but belongs here: it reshapes how the worker VMs are declared before the first `terraform apply`.

---

## ADR-0001 — Split provisioning (Terraform) from configuration (Ansible)
**Status:** Accepted

**Context:** The platform needs both "does this VM exist, with this CPU/disk/IP" (infra existence) and "what's installed and running inside it" (OS/software state). One tool could technically do both — Ansible has cloud provisioning modules, Terraform supports provisioners.

**Decision:** Use Terraform purely for VM provisioning (existence, sizing, IPs, cloning from template) and Ansible purely for configuration (OS prep, k3s install). Never blur the line by using Terraform provisioners to configure, or Ansible to manage infra lifecycle.

**Consequences:** Two tools, two mental models, two different notions of state (Terraform's tracked state file vs. Ansible's stateless idempotent reruns) — more moving parts than a single tool, but each does the job it's actually good at. `terraform apply` converges infra without touching the OS; `ansible-playbook` reruns configuration without recreating VMs.

## ADR-0002 — Pin versions deliberately across the stack
**Status:** Accepted

**Context:** Homelab tooling (k3s, Helm charts, cert-manager, cloudflared, Terraform providers) moves fast; "latest" silently changes what "rerun this guide" produces from one day to the next.

**Decision:** Pin an explicit version everywhere (k3s, `bpg/proxmox`, the Traefik chart, cert-manager, etc.) and treat bumping a version as a conscious, explicit action — check the releases page first, never silently float to `latest`.

**Consequences:** Reproducibility means "the version that was actually tested." Costs a small amount of manual upkeep in exchange for never being surprised by an untested version landing mid-rebuild.

## ADR-0003 — Defer secrets management for v1
**Status:** Accepted

**Context:** SOPS + age scaffolding (`.sops.yaml`, `secrets/`, a worked-example encrypted Secret manifest) was set up alongside the rest of the platform early on, but v1's actual goal is getting the base cluster running end to end, not building out secrets tooling.

**Decision:** Remove the SOPS/age scaffolding entirely for now. Kubernetes Secrets are created imperatively via `kubectl create secret` and are not committed to Git in any form, encrypted or otherwise, until secrets management is picked back up as its own deliberate follow-on.

**Consequences:** One less moving part while learning the core platform. Means every fresh clone or rebuild requires re-entering secret values by hand until SOPS (or ESO/Vault) is reintroduced — tracked as its own item in the v2.0 milestone.

## ADR-0004 — Automate Proxmox host housekeeping and the Tailscale host join via Ansible
**Status:** Accepted

**Context:** Phase 1 (disable the enterprise repo, enable no-subscription, `apt full-upgrade`) and Phase 12 Part A (join `pve-dell` to the tailnet) were both originally documented in GUIDE.md as manual, copy-pasted shell commands.

**Decision:** Codify both as Ansible, targeting a new `[proxmox_hosts]` inventory group, kept in a separate playbook (`proxmox-host.yml`) from the k3s cluster's `site.yml` — different blast radius, different target (the hypervisor, not a k3s node).

**Consequences:** A host rebuild no longer means re-typing commands by hand; reruns are idempotent. Introduces a second playbook and inventory group to maintain alongside the k3s one.

## ADR-0005 — Split Proxmox host automation into two separate roles/playbooks
**Status:** Superseded by ADR-0006

**Context:** Repo/apt housekeeping (Phase 1) and joining the tailnet (Phase 12 Part A) are different concerns with different rerun cadence and different secret requirements — the Tailscale join needs a fresh auth key each time from scratch; apt housekeeping needs nothing.

**Decision:** Create two independent roles (`proxmox_host`, `tailscale_host`) and two playbooks (`proxmox-host.yml`, `tailscale-host.yml`).

**Consequences:** Clean separation of concerns, but doubles the number of near-trivial files (two one-task-file roles, two nearly identical playbooks) for what's ultimately automation against one target host — more directory overhead than the content actually justified.

## ADR-0006 — Merge Proxmox host automation into one role with tagged task files
**Status:** Accepted

**Context:** ADR-0005's two-role split added directory overhead without a real benefit once it became clear Ansible tags could give the same "run independently" property from within a single role.

**Decision:** Merge both concerns into one role (`proxmox_host`) with two task files (`repos.yml`, `tailscale.yml`) imported from `main.yml`, each under its own tag (`repos`, `tailscale`). Run selectively with `--tags repos` or `--tags tailscale`.

**Consequences:** One role, one playbook, the same independent-execution property as the two-role split, less to navigate. This is the shape actually run against `pve-dell`.

## ADR-0007 — Inventory must use portable, resolvable connection targets
**Status:** Accepted

**Context:** `ansible_host=pve-dell` initially relied on a personal `~/.ssh/config` `Host pve-dell` alias to resolve at all — meaning the committed inventory silently only worked from one specific machine, and even then pointed at the wrong SSH key for that host (the k3s VM key, not the Proxmox root key).

**Decision:** Set `ansible_host` to the real, resolvable IP address directly in `inventory.ini`, and set `ansible_ssh_private_key_file` to the key actually authorized for that host — independent of any one machine's personal SSH client configuration.

**Consequences:** The inventory now works identically from any machine holding the right key, matching the "reproducible, minimal manual intervention" goal already stated for the rest of this repo. Caught only because it was flagged before the first real run — worth remembering as a class of bug (see [[Platform Concepts]]).

## ADR-0019 — Workers as a `for_each` map; control plane as a singleton
**Status:** Accepted

**Context:** The original `main.tf` declared `k3s-server-1` and `k3s-worker-1` as two hand-written resource blocks. Adding a worker meant copy-pasting a whole block — and the stated intent ("rerun the worker one to add more workers with new params") revealed a mental-model gap worth encoding structurally: Terraform reconciles the whole directory against one state; there is no per-file apply.

**Decision:** Split into `server.tf` (a deliberate singleton — the control plane is architecture, not a scaling group) and `workers.tf` (one resource block expanded over a `map(object)` variable with `for_each`). Adding a worker = adding an entry to the `workers` map in `terraform.tfvars` and re-applying; the map key doubles as the VM name and the Terraform state key. Done before the first `apply`, so no state migration (`moved` blocks) was needed.

**Consequences:** `terraform plan` after adding an entry shows exactly `1 to add` with existing VMs untouched — the repeatable-workers workflow without duplicated blocks. Worker config drift between entries is now impossible to introduce by editing one copy of a pasted block. Each new worker still needs a matching Ansible inventory entry under `[k3s_agent]`; Terraform and Ansible remain deliberately separate (ADR-0001), so nothing auto-generates that. Removing a map entry destroys that VM on the next apply — the map is authoritative, which is the point, but worth knowing before deleting lines.
