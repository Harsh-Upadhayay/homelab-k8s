# v0.1 — Foundation (ADR-0001 … ADR-0007, ADR-0019 … ADR-0024)

> Back to [[Homelab Learning Map]] · Next [[v0.2 - Cluster Bootstrap]] · Index [[README|ADR index]]

Foundation covers the Proxmox host itself and the tooling boundary that provisions everything above it — the cluster doesn't exist yet at this stage, so these decisions are about repo hygiene and tool boundaries rather than cluster architecture. Provisioning and configuration are kept as two separate tools with two separate mental models (ADR-0001), versions are pinned everywhere on principle (ADR-0002), and secrets management is explicitly pushed out of v1's scope rather than half-built (ADR-0003). The Proxmox-host automation itself went through a real reversal worth keeping visible: it started as two Ansible roles, got merged into one once tags proved sufficient (ADR-0005, superseded by ADR-0006) — and a portability bug in the inventory (a personal SSH alias standing in for a real address) got caught and fixed (ADR-0007).

Provisioning shape landed late in the milestone, once the real host's capacity was known: a `for_each` workers refactor was tried and rejected (ADR-0019), the VMs take nearly the whole box under a strict no-RAM-overcommit rule (ADR-0020), and the future distributed-storage plan got baked into the disk layout — dedicated data disks and a second worker — while Longhorn itself stays a v2.0 install (ADR-0021). The milestone closes with the host's one hard physical constraint on the record: the laptop's internal NVMe (Windows + personal data) is strictly off-limits, and everything runs from the external USB SSD (ADR-0022). The first real `terraform apply` immediately surfaced two more gaps in ADR-0004's own automation, one after the other — storage allocation (ADR-0023) and SDN network attach (ADR-0024) both needed their own narrowly-scoped ACL grants before the cluster's three VMs would actually clone successfully.

---

## ADR-0001 — Split provisioning (Terraform) from configuration (Ansible)
**Status:** Accepted

**Context:** The platform needs both "does this VM exist, with this CPU/disk/IP" (infra existence) and "what's installed and running inside it" (OS/software state). One tool could technically do both — Ansible has cloud provisioning modules, Terraform supports provisioners.

**Decision:** Use Terraform purely for VM provisioning (existence, sizing, IPs, cloning from template) and Ansible purely for configuration (OS prep, k3s install). Never blur the line by using Terraform provisioners to configure, or Ansible to manage infra lifecycle.

**Consequences:** Two tools, two mental models, two different notions of state (Terraform's tracked state file vs. Ansible's stateless idempotent reruns) — more moving parts than a single tool, but each does the job it's actually good at. `terraform apply` converges infra without touching the OS; `ansible-playbook` reruns configuration without recreating VMs.

## ADR-0002 — Pin versions deliberately across the stack
**Status:** Accepted

**Context:** Homelab tooling (k3s, Helm charts, cert-manager, cloudflared, Terraform providers) moves fast; "latest" silently changes what "rerun this guide" produces from one day to the next.

**Decision:** Pin an explicit version everywhere (k3s, `bpg/proxmox`, the Traefik chart, cert-manager, etc.) and treat bumping a version as a conscious, explicit action — check the releases page first, never silently float to `latest`.

**Consequences:** Reproducibility means "the version that was actually tested." Costs a small amount of manual upkeep in exchange for never being surprised by an untested version landing mid-rebuild.

## ADR-0003 — Defer secrets management for v1
**Status:** Accepted

**Context:** SOPS + age scaffolding (`.sops.yaml`, `secrets/`, a worked-example encrypted Secret manifest) was set up alongside the rest of the platform early on, but v1's actual goal is getting the base cluster running end to end, not building out secrets tooling.

**Decision:** Remove the SOPS/age scaffolding entirely for now. Kubernetes Secrets are created imperatively via `kubectl create secret` and are not committed to Git in any form, encrypted or otherwise, until secrets management is picked back up as its own deliberate follow-on.

**Consequences:** One less moving part while learning the core platform. Means every fresh clone or rebuild requires re-entering secret values by hand until SOPS (or ESO/Vault) is reintroduced — tracked as its own item in the v2.0 milestone.

## ADR-0004 — Automate Proxmox host housekeeping and the Tailscale host join via Ansible
**Status:** Accepted

**Context:** Phase 1 (disable the enterprise repo, enable no-subscription, `apt full-upgrade`) and Phase 12 Part A (join `pve-dell` to the tailnet) were both originally documented in GUIDE.md as manual, copy-pasted shell commands.

**Decision:** Codify both as Ansible, targeting a new `[proxmox_hosts]` inventory group, kept in a separate playbook (`proxmox-host.yml`) from the k3s cluster's `site.yml` — different blast radius, different target (the hypervisor, not a k3s node).

**Consequences:** A host rebuild no longer means re-typing commands by hand; reruns are idempotent. Introduces a second playbook and inventory group to maintain alongside the k3s one.

## ADR-0005 — Split Proxmox host automation into two separate roles/playbooks
**Status:** Superseded by ADR-0006

**Context:** Repo/apt housekeeping (Phase 1) and joining the tailnet (Phase 12 Part A) are different concerns with different rerun cadence and different secret requirements — the Tailscale join needs a fresh auth key each time from scratch; apt housekeeping needs nothing.

**Decision:** Create two independent roles (`proxmox_host`, `tailscale_host`) and two playbooks (`proxmox-host.yml`, `tailscale-host.yml`).

**Consequences:** Clean separation of concerns, but doubles the number of near-trivial files (two one-task-file roles, two nearly identical playbooks) for what's ultimately automation against one target host — more directory overhead than the content actually justified.

## ADR-0006 — Merge Proxmox host automation into one role with tagged task files
**Status:** Accepted

**Context:** ADR-0005's two-role split added directory overhead without a real benefit once it became clear Ansible tags could give the same "run independently" property from within a single role.

**Decision:** Merge both concerns into one role (`proxmox_host`) with two task files (`repos.yml`, `tailscale.yml`) imported from `main.yml`, each under its own tag (`repos`, `tailscale`). Run selectively with `--tags repos` or `--tags tailscale`.

**Consequences:** One role, one playbook, the same independent-execution property as the two-role split, less to navigate. This is the shape actually run against `pve-dell`.

## ADR-0007 — Inventory must use portable, resolvable connection targets
**Status:** Accepted

**Context:** `ansible_host=pve-dell` initially relied on a personal `~/.ssh/config` `Host pve-dell` alias to resolve at all — meaning the committed inventory silently only worked from one specific machine, and even then pointed at the wrong SSH key for that host (the k3s VM key, not the Proxmox root key).

**Decision:** Set `ansible_host` to the real, resolvable IP address directly in `inventory.ini`, and set `ansible_ssh_private_key_file` to the key actually authorized for that host — independent of any one machine's personal SSH client configuration.

**Consequences:** The inventory now works identically from any machine holding the right key, matching the "reproducible, minimal manual intervention" goal already stated for the rest of this repo. Caught only because it was flagged before the first real run — worth remembering as a class of bug (see [[Platform Concepts]]).

## ADR-0019 — Workers as a for_each map
**Status:** Rejected

**Context:** With more workers planned eventually, two hand-written VM resource blocks looked like duplication. A refactor split the Terraform into `server.tf` (singleton) plus `workers.tf` (one resource expanded over a `map(object)` variable), so adding a worker becomes a map entry in tfvars.

**Decision:** Tried, then reverted (commits `007f965` → `5038cf5`) in favor of explicit, separately named resource blocks in `main.tf`. Recorded here after the fact — the revert originally scrubbed this number from the index, but ADRs are append-only: rejected experiments stay on the record. (The revert also accidentally deleted `main.tf` without restoring it, leaving zero resource blocks — caught before the first apply.)

**Consequences:** At two-to-three identically-specced workers, explicit blocks read more clearly than loop indirection, at the cost of ~55 duplicated lines per worker. At 3+ workers the map starts earning its keep — revisit then; it's a one-commit refactor plus `terraform state mv` if done after apply (done before first apply, it's free).

## ADR-0020 — Near-full host allocation: overcommit CPU, never RAM
**Status:** Accepted

**Context:** `pve-dell` (14 threads, 30GiB usable RAM, 816GiB NVMe thin pool) exists solely to host this cluster's VMs — the goal is to hand them as much of the box as safely possible. But VM RAM is real memory: if combined usage exceeds physical RAM, the host OOM-killer kills a whole VM, and if that VM is `k3s-server-1`, etcd — and the cluster — dies with it. vCPUs, by contrast, are schedulable threads, not reservations.

**Decision:** Allocate 16 vCPU across 14 threads (mild overcommit — k8s load is bursty) and 24GiB of 30GiB RAM (server 6GiB, workers 9GiB each), leaving a ~5GiB host reserve. Disks are thin-provisioned: 680GB declared of 816GB, blocks consumed only as written.

**Consequences:** The cluster gets effectively the whole box while etcd can never meet the OOM-killer. Worker RAM can be raised later (a `terraform apply` away) once real usage is observed on the host under load; CPU contention worst-case is slowdown, never breakage.

## ADR-0021 — Distributed storage direction: dedicated per-worker data disks now, Longhorn later
**Status:** Accepted

**Context:** The long-term plan is adding physical nodes, each contributing its own storage to one pool that any pod on any node can mount — storage-node-agnostic workloads. That is distributed block storage; in the k3s ecosystem the natural fit is Longhorn (lightweight, Rancher-native, volumes replicated across nodes and mounted via CSI/iSCSI from anywhere). Ceph/Rook is the heavier alternative, wrong-sized for this stage.

**Decision:** Shape the provisioning for it now, install Longhorn later as its own additive phase (tracked in v2.0). Concretely now: (1) every worker gets a dedicated second disk (`scsi1`, 250GB thin, raw and empty — not part of the template clone), formatted ext4 with label `k3s-data` and mounted at `/var/lib/longhorn` by the `k3s_agent` role; (2) a second worker (`k3s-worker-2`, `.23`) exists so rescheduling and replica behaviour are actually observable — with one worker, "the pod can run anywhere" is untestable.

**Consequences:** OS I/O and storage I/O stay on separate disks; wiping/rejoining storage is independent of OS reinstall; future physical nodes arrive with the same symmetric layout (OS disk + data disk) and just join the pool. Honest caveat, recorded deliberately: both workers' disks live on the same physical device (see ADR-0022), so replication between them is a *topology simulation*, not durability — real redundancy begins when real machines arrive. Longhorn's control-plane taint interaction is already correct (it runs on workers only).

## ADR-0022 — The internal NVMe is strictly off-limits; everything lives on the external SSD
**Status:** Accepted

**Context:** `pve-dell` is a laptop. Its internal 1TB NVMe (Samsung PM9C1a) holds a Windows install and personal data. Proxmox was installed from a flash drive onto a separate external 1TB USB SSD (`SSD-PUT/N`), and that SSD carries everything: EFI boot, root, swap, and the entire `pve` LVM volume group backing both storage pools. Verified directly on the host: the only LVM physical volume is `/dev/sda3` (the external SSD), and `/etc/pve/storage.cfg` references nothing else — the NVMe is visible to the kernel but untouched by any Proxmox configuration.

**Decision:** The internal NVMe must never be added as a storage pool, LVM physical volume, mount, or passthrough target — under any circumstances, including "there's free space" or "etcd would be faster there." The external SSD is the only working storage. Capacity and performance grow later by adding physical nodes with their own disks (which is the plan ADR-0021 already shapes for), never by touching this disk.

**Consequences:** etcd — and everything else — runs on a USB-attached SSD, not internal NVMe. USB bridges have higher and less predictable fsync latency, so the earlier "non-negotiable NVMe for the control plane" assumption is consciously downgraded to "best disk available under the constraint"; if etcd ever shows leader-election flakiness or slow-apply warnings, the disk is the first suspect, and that's an accepted tradeoff rather than a bug. The USB link is also a single point of failure for the whole host (mitigated: the `uas` driver pins the disk against USB autosuspend, and lid-close suspend is disabled — the latter now codified in the `proxmox_host` role's `laptop` tag). The laptop's two-disk reality also explains the deliberately conservative thin-pool headroom in ADR-0020.

## ADR-0023 — terraform@pve needs a storage-scoped ACL, not just PVEVMAdmin
**Status:** Accepted

**Context:** ADR-0004 automated the `terraform@pve` token bootstrap with a single ACL grant, `pveum aclmod / -user terraform@pve -role PVEVMAdmin` — mirroring GUIDE.md's original by-hand instructions. The first real `terraform apply` failed on all three VMs: `HTTP 403 - Permission check failed (/storage/local-lvm, Datastore.AllocateSpace)`. Proxmox keeps VM privileges (`VM.Clone`, `VM.Config.*`, ...) and storage privileges (`Datastore.Allocate`, `Datastore.AllocateSpace`, ...) in two entirely separate buckets — `PVEVMAdmin` contains none of the latter, confirmed via `pveum role list`. Cloning a VM has to allocate space on the target datastore, which needs `Datastore.AllocateSpace` regardless of how much VM-level permission the token already has.

**Decision:** Grant a second ACL, scoped to the specific storage path rather than `/`: `pveum aclmod /storage/local-lvm -user terraform@pve -role PVEDatastoreUser` (which carries exactly `Datastore.AllocateSpace` + `Datastore.Audit` — not the broader `PVEDatastoreAdmin`, since Terraform here only ever clones onto existing storage, never creates pools or uploads templates). Codified as an idempotent, independently-checked step in `terraform_token.yml` — checked directly against `pveum acl list` rather than piggybacked on the user-existence check, specifically so it self-heals a host that already bootstrapped without it (as `pve-dell` had).

**Consequences:** `terraform apply` can now actually allocate disk space. The token stays least-privilege: VM operations at `/`, storage operations scoped to only the one pool actually in use — matching the same reasoning as the token's existence in the first place (ADR-0004, scoped token vs. root password). Worth remembering as a general Proxmox ACL lesson: a role that "sounds" comprehensive (VM*Admin*) does not imply cross-cutting resources like storage are included — check `pveum role list` rather than assuming.

## ADR-0024 — terraform@pve also needs an SDN-scoped ACL for network attach
**Status:** Accepted

**Context:** With ADR-0023's storage fix applied, the very next `terraform apply` attempt hit a *third* separate privilege bucket: `HTTP 403 - Permission check failed (/sdn/zones/localnetwork/vmbr0, SDN.Use)` on all three VMs, at the network-device-attach step of cloning. `pvesh get /cluster/sdn/zones` returns `[]` — no SDN zones or vnets are actually configured on this host — yet Proxmox VE still evaluates every bridge attachment against an implicit `/sdn/zones/localnetwork/<bridge>` path for permission purposes. Confirmed via `pveum role list`: `SDN.Use` lives only in `PVESDNUser`/`PVESDNAdmin`/broad admin roles, none of which `PVEVMAdmin` or `PVEDatastoreUser` include.

**Decision:** Grant a third ACL, again scoped to exactly what's used: `pveum aclmod /sdn/zones/localnetwork/vmbr0 -user terraform@pve -role PVESDNUser` (carries `SDN.Audit` + `SDN.Use` — not `PVESDNAdmin`'s additional `SDN.Allocate`, since Terraform never creates SDN zones/vnets here, only attaches to the existing bridge). Implemented in `terraform_token.yml` reusing the `pveum_acls` fetch already made for ADR-0023's check — no second `pveum acl list` call needed, just another `selectattr` chain and conditional grant against the same data.

**Consequences:** Three independent privilege buckets now confirmed the hard way for one VM-clone operation: `VM.*` (ADR-0004), `Datastore.*` (ADR-0023), `SDN.*` (this one) — cloning with cloud-init networking touches all three regardless of whether SDN is "in use" in any meaningful sense. `terraform apply` succeeded immediately after this grant: 3 VMs created, all reachable at their static IPs (`k3s-server-1`/`.21`, `k3s-worker-1`/`.22`, `k3s-worker-2`/`.23`). General lesson reinforced: when bootstrapping a scoped Proxmox token for a new kind of operation, expect to discover its full privilege surface by hitting each 403 in turn rather than being able to look it up in one place — Proxmox's own role-to-privilege mapping (`pveum role list`) is the fastest way to close each gap as it appears.
