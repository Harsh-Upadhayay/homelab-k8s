## Helm

**Helm is a package manager for Kubernetes**, the same relationship `apt` has to `.deb` files. The problem it solves: a single "application" (here, the Tailscale Operator) is never one YAML file — it's a Namespace, a Deployment, RBAC objects, CRDs, and a Secret, all needing to be created, upgraded, and removed together as one unit.

**A chart is the package**: `Chart.yaml` (metadata), `values.yaml` (defaults), `templates/` (Kubernetes manifests written as Go templates referencing those values — the same *idea* as Ansible's Jinja2 `template` module, rendering a final config from a template + values, just scaled up to a whole set of interconnected objects instead of one file). A **chart repository** (`helm repo add`) is an HTTP index of available charts, same relationship an apt repo has to `.deb` files.

**`helm upgrade --install` renders, then applies.** It layers overrides on defaults (CLI `--set-string` beats `--values <file>` beats the chart's own defaults), feeds the result through the template engine to produce final, literal YAML (inspectable in isolation via `helm template`, no cluster contact), then applies it to the cluster like `kubectl apply -f` would for many objects at once. It records what it did as a **release** — stored as a `Secret` inside the cluster itself — which is what makes `helm upgrade` a real diff against known state rather than a blind re-apply, and `helm uninstall` able to cleanly remove every object a release created. Visible directly: `sh.helm.release.v1.tailscale-operator.v1` and `.v2` appeared as real Secrets after we fixed a bad OAuth client and re-ran `helm upgrade` — one Secret per revision (`k8s/tailscale/operator-values.yaml`, ADR-0025).

**Release name vs. chart reference are two different things that can look identical.** `helm upgrade --install tailscale-operator tailscale/tailscale-operator` — the first `tailscale-operator` is the release name *we chose*; the second is `<repo-alias>/<chart-name>`. They match here only because that's the sensible name to pick, not because they're the same field.

**`--wait` blocks until resources report ready, not just accepted** — same "submission isn't completion" lesson as Ansible's `wait_for`, Helm's version of it.

**Helm repos are local to the machine's CLI, not synced anywhere.** `helm repo add`/`helm repo update` write to that machine's own local cache (`~/.cache/helm`, `~/.config/helm`) — not to the cluster, and not to any other machine that happens to also run `helm`. Running the repo-add/update steps from one machine (a MacBook, over the tailnet) and the actual `helm install` from a different one (this Linux workstation, which has `k8s/traefik/values.yaml` on disk) failed in an uninformative way: no error, just no pods, no Service, an empty `helm list -n traefik` — because the second machine's Helm had simply never heard of the `traefik/traefik` chart reference it was asked to resolve. Fixed by running `helm repo add`/`update` again, locally, on whichever machine actually runs the install.

**A chart's values schema isn't stable across chart versions, even between routine releases.** The Traefik chart moved `ports.websecure.tls` to `ports.websecure.http.tls` between versions — strict enough that `helm install` refused outright (`additional properties 'tls' not allowed`) — and separately moved `service.type` to `service.spec.type`, a change the schema does *not* enforce, so the old key was silently ignored and the Service defaulted to the chart's own `LoadBalancer` default with zero error, only visible by actually reading `kubectl get svc`'s `TYPE` column rather than trusting the exit code. `helm show values <repo>/<chart>` prints the live, current schema straight from the chart — worth diffing a repo's committed values file against it before trusting that file still means what it meant when it was written (ADR-0027, `k8s/traefik/values.yaml`).

**Helm deliberately does not manage CRD upgrades or deletions.** CRDs ship in a chart's separate `crds/` directory (not `templates/`); `helm install` applies them the very first time, but `helm upgrade`/`helm uninstall` never touch them again on purpose — a routine chart change silently rewriting or deleting a cluster-wide schema is considered too dangerous to automate. The convention this leads to (Traefik included): manage CRDs explicitly, separately from the chart's own lifecycle. `helm show crds <chart>` extracts just that `crds/` content, with zero cluster contact — no install, no templating of anything else in the chart — ready to be piped into `kubectl apply` on its own.

**Server-side apply exists because client-side apply has a hard size ceiling.** Client-side `kubectl apply` (the default) stores the entire last-applied config in an annotation on the object, capped at roughly 256KB — Traefik's CRDs, with their full embedded OpenAPI schemas, are large enough to blow past that and fail outright. `--server-side` instead sends the object to the API server and lets *it* compute the diff and track field ownership per "manager" (the tool that last set that field) — no local annotation, no size limit.

**`--force-conflicts` matters the moment the same apply command runs a second time.** Server-side apply refuses to silently overwrite a field some other manager currently owns. A first install rarely trips this, but re-running the identical `helm show crds | kubectl apply --server-side` command later — e.g. after a chart upgrade changes a CRD field — would otherwise demand manual conflict resolution. `--force-conflicts` declares that this command is always the intended source of truth for these objects, so it should simply take ownership of whatever it specifies.

**OCI charts are a different distribution mechanism than a Helm HTTP repo — and immune to the per-machine repo problem.** A traditional repo is an HTTP server hosting an `index.yaml` + packaged `.tgz` files, which `helm repo add` caches *locally per machine* (the exact reason the Traefik install failed when repo-add ran on one box and install on another). An **OCI chart** is instead pushed as an artifact into a container registry — same registry tech that stores container images — and `helm install oci://quay.io/jetstack/charts/cert-manager` pulls it *directly by reference*: no `helm repo add`, no local index, no `helm repo update` step at all, from any machine every time. cert-manager's own docs now treat the OCI chart as the source of truth ("published immediately upon release") and the legacy `jetstack/cert-manager` HTTP repo as secondary. Lesson beyond the mechanics: a repo continuing to *resolve* (`helm search repo` still finds the chart) doesn't mean it's still the maintainer's *recommended* path — that's a judgment the docs make, not something the tooling can tell you.

**Two charts, two opposite conventions for the same CRD-lifecycle problem — don't assume one generalizes.** Both Traefik and cert-manager solve "Helm won't safely upgrade/delete CRDs on its own," but land on opposite mechanisms. Traefik keeps CRDs *outside* the chart — you `helm show crds` and `kubectl apply` them yourself, fully decoupled from the release. cert-manager keeps CRDs *inside* the chart, gated behind a values flag `crds.enabled=true` (plus `crds.keep=true` to annotate them so `helm uninstall` won't garbage-collect them — and take every Certificate/Issuer with them). Neither is more correct; it's a per-maintainer choice, discoverable only by reading each project's own docs/values, never inferable from the previous chart.

**Deprecation lives inside a chart's own values schema, not just the Kubernetes API.** cert-manager's values show `installCRDs` (deprecated) sitting right beside `crds.enabled`, with a comment pointing old→new. A chart's `values.yaml` is itself a versioned interface with a migration path — old field names linger as aliases/no-ops for backward compatibility, exactly like a deprecated field in a real API version. Same instinct as `kubectl explain`: read the live `helm show values <chart>` rather than trusting a remembered field name, since the "current" field quietly shifts chart-version to chart-version.

**With Helm you write only `values.yaml` — Helm renders and owns the actual objects.** `k8s/traefik/` has no Service or Deployment file, yet the `traefik` Service/Deployment/ServiceAccount/RBAC all exist — rendered by the chart's `templates/` from our values (merged over chart defaults) and applied by `helm install`. Proof: the live Service carries labels we never wrote — `helm.sh/chart: traefik-41.0.1`, `app.kubernetes.io/managed-by: Helm`. See the full rendered set with `helm get manifest <release> -n <ns>`. This is a different authoring model from the hand-written manifests in this repo (cloudflared, the ClusterIssuers, the example app, the Traefik *dashboard* Service) which we write in full and `kubectl apply` ourselves — a split GitOps (Argo CD, v2.0) will later reconcile both sides of.

## Core objects: Namespace, Deployment, ServiceAccount

**A Namespace is a naming/isolation boundary within one cluster** — `kubectl create namespace tailscale` before the Helm install exists so this chart's objects don't collide with `traefik`'s or `cert-manager`'s later. Some charts create their own namespace (`--create-namespace`); this one expects it pre-created.

**A Deployment manages a pod running the actual application** — here, the Operator's own controller process. Nodes were the only Kubernetes object this project had touched directly before this; a Deployment is the first "workload" object.

**A ServiceAccount is an identity *for a pod*, not a human** — every pod runs *as* some ServiceAccount, and RBAC bound to that ServiceAccount determines what its own process can do when it calls back into the API server (which the Operator does constantly, via `client-go`, watching Services/Ingresses/its own CRDs). Confirmed via `kubectl describe pod`: `Service Account: operator`.

## The object model itself: what "a Kubernetes object" even is

**Every Kubernetes object is the same fundamental thing — a record in etcd describing desired state, that some controller works to make real.** A `Pod` isn't a running container; it's a *record saying* "I want this container running." A `Deployment` is a record saying "keep N of these pods alive." A `Secret` is a record holding data. None of them are the live thing — they're declared intent, plus a controller reconciling reality toward it. `kubectl apply` writes the record (via the API server, into etcd); `kubectl get` reads it back. This is the single core idea the whole system is built on, and it's identical across every kind.

**Every object shares one skeleton**, built-in or custom: `apiVersion` (which API group+version defines this kind), `kind`, `metadata` (name/namespace/labels), `spec` (the desired state *you* write), `status` (the actual state the *controller* fills in). `Pod`, `Deployment`, `Secret`, `ClusterIssuer`, `Certificate` — all fit this exact mold, which is why they all feel the same to `kubectl`.

**Built-in kinds vs. CRD-added kinds — the `apiVersion` group is the tell.** `Pod`/`Secret` are `v1`, `Deployment` is `apps/v1` — the core/apps groups the API server knows natively from birth. `ClusterIssuer`/`Certificate` are `cert-manager.io/v1`; `Connector`/`ProxyClass` are Tailscale's — that non-core group name is the fingerprint of a kind added by an installed operator's CRDs. A vanilla cluster has literally never heard of these until the CRD is installed.

**A CRD makes an object *storable*; a controller makes it *meaningful* — two separate halves.** Installing cert-manager's CRDs (the `crds.enabled=true` step) teaches the API server the `ClusterIssuer`/`Certificate` nouns: `kubectl get clusterissuer` starts working, the objects store in etcd, indistinguishable from built-ins at the API layer. But the API server only *stores* them — it has no idea what a "Certificate" means. The behavior comes entirely from the **controller** (cert-manager's running pods) watching for those objects and acting. Proof of the split: `kubectl apply` a `Certificate` with the controller absent and it just sits inert in etcd — a declared intent with nobody working to fulfill it. Same lesson the Tailscale operator taught, restated: CRD = vocabulary, controller = behavior.

**Your cluster's set of object types is open, not a fixed ~30.** Every operator installed (Tailscale, cert-manager, later Argo CD / Prometheus / Longhorn) extends *this specific cluster's* vocabulary with its own nouns — two clusters can speak different dialects depending on what's installed. `Pod`/`Deployment`/`Secret` are the words every cluster knows; `ClusterIssuer` is one *this* cluster learned. `kubectl api-resources` prints the full current vocabulary — built-ins and CRD-added kinds together, with a NAMESPACED column — and is the direct way to *see* a CRD install having happened (the new kinds appear in the list afterward).

**Scope is an axis every kind sits on: namespaced vs. cluster-scoped.** `Pod`/`Secret`/`Certificate`/`Issuer` are namespaced — they live inside one namespace and a same-named object in another namespace is a different object (this is why cert-manager writes the cert Secret into `traefik` specifically, where Traefik can mount it). `ClusterIssuer`/`Node`/`ClusterRole`/`Namespace` are cluster-scoped — one global instance, usable everywhere. cert-manager deliberately ships *both* `Issuer` (namespaced) and `ClusterIssuer` (cluster-scoped) as separate kinds so you pick the scope; we use `ClusterIssuer` because the signing config is a shared cluster capability, not something to duplicate per namespace. Exact same namespaced-vs-cluster split already seen in RBAC (`Role` vs `ClusterRole`).

## CRDs and the Operator pattern

**A CRD (Custom Resource Definition) extends the Kubernetes API with a new object kind.** Kubernetes isn't a fixed, closed set of types — anyone can define a new one, and the API server will then let you create/list/watch instances of it exactly like built-in types (`Pod`, `Node`). The Tailscale chart installs CRDs like `Connector` and `ProxyClass` — new vocabulary the cluster now understands.

**The Operator pattern = CRD(s) (or existing resources) + a controller that continuously reconciles real-world state to match declared state.** The Operator's pod *watches* the API for things it cares about (Services requesting `loadBalancerClass: tailscale`, its own CRD instances) and *reacts* by creating/managing real tailnet devices. This watch-and-react loop is the general pattern — cert-manager's `Certificate` objects will work identically later.

## Kubernetes's own permission system: RBAC

**Authentication and authorization are strictly separate systems.** Proven concretely: a second device (a MacBook) got a *real* `403 Forbidden` response from the actual API server — meaning authentication (the proxy correctly identified the caller as a specific Tailscale login) had already fully succeeded; only authorization (does that identity have any permissions) failed. Two independent failure modes: one is a connectivity/identity problem, the other is a policy problem — never confuse which one you're debugging.

**Four RBAC object kinds, two axes.** `Role`/`ClusterRoleBinding` define *what can be done*; `RoleBinding`/`ClusterRoleBinding` grant it *to someone*. The namespace-scoped pair (`Role`, `RoleBinding`) only applies within one namespace; the cluster-scoped pair applies everywhere, and is *required* — not just preferred — for non-namespaced resources like `Node`. `kubectl get nodes` structurally cannot be granted via a `RoleBinding`, no matter what it references, because `Node` doesn't belong to any namespace to bind within (ADR-0026).

**`cluster-admin` is a built-in `ClusterRole`, not something this project defined** — ships with every standard Kubernetes install, `*` verbs on `*` resources in `*` API groups, and is the same role k3s's own `system:masters` group (and therefore the original admin `kubeconfig` from Phase 4–6) is already bound to.

**Only `ServiceAccount` is a real object among the three RBAC subject kinds.** `User` and `Group` aren't Kubernetes resources at all — no `kubectl get users` exists — they're just strings asserted by whatever authentication method produced them, which RBAC trusts without further verification. Here, that string arrives via `Impersonate-User: harshupadhayay906@gmail.com`, set by the Tailscale proxy — standard Kubernetes user impersonation, nothing Tailscale-specific about the mechanism itself. The trust boundary is entirely at the authentication step (Tailscale's own device/login verification), outside Kubernetes proper (`k8s/tailscale/api-server-rbac.yaml`, ADR-0026).

**`roleRef` is immutable after a binding is created.** To point a binding at a different role, delete and recreate it — the API rejects in-place edits to that field.

## Debugging: the standard triage flow

**`kubectl get pods` → `describe` → `logs`/`logs --previous`, in that order, each answering a different question.** Watched live against a real crash-looping pod:
- **`get pods`** shows *that* something's wrong: `READY 0/1`, `STATUS: CrashLoopBackOff`, a climbing `RESTARTS` count.
- **`describe pod`** gives *history and context* — the `Events` list (a timeline of scheduling/pulling/starting/backing-off, oldest to newest — usually the first thing worth reading top to bottom), exit codes, mounted volumes, env vars. It told us the container reached `Started` and died 2 seconds later with `Exit Code: 1` — narrowing *where*, not yet *why*.
- **`logs` / `logs --previous`** hold the actual reason, in the container's own stdout/stderr. `--previous` matters specifically once a container has already restarted: the *current* instance's logs might just be an uninformative fresh startup, while `--previous` preserves what the crashed instance actually said. This is where the real, specific, actionable error (`"creating operator authkey: ... (403)"`) was found — `describe` alone never would have surfaced it (ADR-0025).

**`CrashLoopBackOff` is a specific, meaningful state**, not a generic "broken." It means the container has crashed *repeatedly*, and Kubernetes is deliberately waiting with exponential backoff before retrying — distinct from `Error` (crashed once, not yet backing off), `Pending` (not scheduled/starting), or `ContainerCreating` (still pulling/creating).

## Multi-client cluster access via a proxy

**A kubeconfig doesn't have to contain any real Kubernetes credential.** Tested directly: `tailscale configure kubeconfig <name>` on a machine with zero prior kubeconfig created the entire file from scratch — and the resulting `user.token` field was the literal string `"unused"`. All real authentication happens one layer down, at the network level (the caller's Tailscale/WireGuard identity) — `kubectl` doesn't present a Kubernetes-native credential at all; by the time a request arrives at the proxy, the caller's identity is already established.

**The Tailscale API-server proxy forwards the main API but not the OpenAPI endpoints — so `kubectl get` works and `kubectl apply` breaks.** Applying `cluster-issuer.yaml` through the proxy failed with `failed to download openapi: the server could not find the requested resource`. Root cause: `kubectl apply` downloads the cluster's OpenAPI schema (from `/openapi/v3`) to validate a manifest *before* sending it, and the proxy 404s those paths; `kubectl get` needs no schema, which is why reads had worked fine and masked the gap. Isolated it by reproducing `kubectl apply --dry-run=server` against the *direct* server (`192.168.1.21:6443`) — it succeeded and served `/openapi/v3` perfectly — proving the manifest and cluster were fine and the fault was purely the proxy transport. Two fixes: on a same-LAN machine, use the direct admin kubeconfig (full OpenAPI, no proxy hop needed anyway); through the proxy from a remote machine, add `--validate=false` to skip only the *client-side* schema fetch — safe because the admission webhook still validates server-side (and `--dry-run=server` had already confirmed validity). General lesson: `get` succeeding through a proxy doesn't prove `apply` will — reads and writes exercise different API surfaces.

**The Operator's second mechanism: claiming a `LoadBalancer` Service via `loadBalancerClass: tailscale`.** Distinct from the API-server proxy above — this is how *workloads* (not the API) get a tailnet front door. Any Service with `type: LoadBalancer` + `loadBalancerClass: tailscale` gets claimed by the Operator: it spins up a proxy pod in the `tailscale` namespace (`ts-<hostname>-*`) that joins the tailnet as its *own device*, and the Service's `EXTERNAL-IP` column fills in with the device's Tailscale IP + MagicDNS name (observed live: `100.79.208.52,traefik-internal.egret-pence.ts.net`, claimed in ~30s). The `tailscale.com/hostname` annotation pins the device name — determinism matters here because DNS points at that name (ADR-0029): without the pin, a recreated proxy gets an auto-generated or suffixed name and every CNAME aimed at it breaks. Exercised for Traefik's internal front door in `k8s/traefik/tailscale-service.yaml` — the tailnet counterpart of what cloudflared is for the public path.

## cert-manager: automated TLS via ACME

**The problem cert-manager solves is the certificate *lifecycle*, not just issuance.** A browser only trusts an HTTPS cert signed by a CA already in its trust store; a Let's Encrypt cert lives only 90 days by design. So the real work isn't getting one cert — it's proving domain ownership, obtaining, storing it where the workload can mount it, and **re-doing all of that before every 90-day expiry, forever.** cert-manager is an operator (same CRD-plus-reconciling-controller shape as the Tailscale operator) that turns this into a declarative fact: you write a `Certificate` object, and its controller obtains, stores, and auto-renews the cert with no human in the loop. Exercised end to end in Phase 10 — a real cert now sits in the `internal-wildcard-tls` Secret, issuer `O = Let's Encrypt, CN = YR1`, valid to Oct 2026.

**cert-manager doesn't sign certs — Let's Encrypt does; cert-manager orchestrates the proof.** The signer is Let's Encrypt (free, automated, browser-trusted-root CA); the protocol is **ACME**; cert-manager is an ACME *client*. Holding that split straight is what makes the Cloudflare-token role (below) make sense.

**Staging vs. prod are the same CA at two endpoints — issuer choice is one field.** `letsencrypt-staging` and `letsencrypt-prod` differ only in their `acme.server` URL. Staging has near-unlimited rate limits but an untrusted root (a sandbox); prod is genuinely rate-limited (e.g. ~5 duplicate certs/week) but real. Workflow proven in Phase 10: issue against staging first, confirm `Ready=True`, then flip `issuerRef.name` → prod and re-apply — cert-manager saw the issuer change and re-issued, and `openssl x509 -issuer` went from a `(STAGING)` issuer to `O = Let's Encrypt`. Two named `ClusterIssuer`s make that switch a one-line edit rather than a credential swap.

**`privateKeySecretRef` is the ACME *account* key, not the certificate's key.** Your relationship with Let's Encrypt is itself an account identified by a keypair; cert-manager generates it on first use and stores it in that Secret. Staging and prod are separate accounts with separate keys (`-staging-account-key` vs `-prod-account-key`).

**An Issuer going `Ready=True` proves ACME account registration only — NOT that the DNS-01 solver works.** Registering the account (creating + storing the account key) is all that's validated when the `ClusterIssuer` reports Ready. The Cloudflare token is exercised for the *first time* only when an actual `Certificate` requests issuance. So a green issuer is a necessary-but-insufficient signal; the real end-to-end proof is a `Certificate` reaching `Ready=True`.

**HTTP-01 vs DNS-01, and why DNS-01 was mandatory here.** HTTP-01 proves control by serving a token at `http://<domain>/.well-known/acme-challenge/...` — which requires the domain to be **publicly reachable on port 80**. DNS-01 proves control by writing a TXT record at `_acme-challenge.<domain>` that Let's Encrypt then queries over **public DNS**. DNS-01 was required for two independent reasons, either sufficient: (1) these services are internal-only (reached over Tailscale — there is no public port 80 for HTTP-01 to hit), and (2) Let's Encrypt refuses to issue **wildcards** over HTTP-01 at all. The elegant inversion that makes internal-service TLS work: you prove control of the *public DNS zone*, not reachability of the *private service* — public proof, private target.

**What the Cloudflare API token actually does: write one TXT record, then delete it.** Not create subdomains, not sign anything. During issuance cert-manager uses the token to create `_acme-challenge.in.neovara.uk TXT = <value LE asked for>`, LE queries public DNS and sees it (proving zone control), LE signs the cert, and cert-manager deletes the temporary record. That's the entire use — which is exactly why the `"Edit zone DNS"` scope is right and nothing broader is needed, and why the zone (`neovara.uk`) must be authoritatively on Cloudflare so LE's public query resolves. The token has no part in signing the cert (LE does that, cryptographically) nor in making `grafana.in.neovara.uk` *resolve* to anything (a separate DNS/MagicDNS concern, still unbuilt).

**One wildcard cert covers every future internal service.** `*.in.neovara.uk` issued once serves `grafana.in.…`, `argocd.in.…`, `traefik.in.…`, and anything added later — no per-service reissue. The tradeoffs paid for that convenience: a larger blast radius if the key leaks, and the hard requirement of DNS-01 (wildcards can't use HTTP-01). Chosen deliberately for a single-operator homelab where the convenience clearly outweighs the blast-radius concern.

**Cluster Resource Namespace: a `ClusterIssuer`'s input secrets and its output cert flow in opposite directions.** Because a `ClusterIssuer` is cluster-scoped, every Secret it *reads* — the ACME account key and the DNS-01 credential (`cloudflare-api-token`) — is resolved from one fixed **Cluster Resource Namespace** (default = cert-manager's install namespace, `cert-manager`; confirmed via the chart's `clusterResourceNamespace: ""` → release namespace), regardless of which `Certificate` triggers issuance. But the *issued* cert Secret is *written* into the `Certificate`'s **own** namespace (`traefik`), so the consuming pod can mount it. The trap is assuming both live together: input credentials centralize in `cert-manager`, the output cert decentralizes to `traefik` — opposite directions, validated from live docs before running (`GUIDE.md` Phase 10).

**Why this stack over the alternatives.** Self-signed / private CA — nothing trusts it without hand-installing a root everywhere. Manual or commercial certs — cost plus manual renewal, the exact toil being eliminated. Certbot on a host — same ACME idea but not Kubernetes-native, no namespaced-Secret delivery to the workload. Cloudflare Origin CA certs — trusted only by Cloudflare's edge, useless for internal services a browser reaches directly. Relying on Cloudflare edge TLS (what the *public* path does, ADR-0015) — internal services never traverse the edge, so they'd get no TLS at all. HTTP-01 — can't reach internal services and can't do wildcards. The chosen combination (cert-manager + Let's Encrypt + DNS-01 via Cloudflare + a wildcard) is the only one that is simultaneously free, auto-renewing, browser-trusted, functional for publicly-unreachable names, and coverage-complete for all future internal subdomains.

## Services, ClusterIP, and cluster DNS (CoreDNS)

**A Service's DNS name is `<service>.<namespace>.svc.<cluster-domain>`.** For `traefik.traefik.svc.cluster.local`: `traefik` (service) · `traefik` (namespace) · `svc` (it's a Service, vs the parallel `pod` form) · `cluster.local` (k3s's default cluster domain). k3s ships **CoreDNS** as the resolver; every pod's `/etc/resolv.conf` points its `nameserver` at the `kube-dns` Service ClusterIP (`10.43.0.10` here — the *Service* keeps the name `kube-dns` for backward-compat even though the software behind it is CoreDNS). CoreDNS's `kubernetes` plugin watches the API and answers these names from live Service/EndpointSlice state (`GUIDE.md` Phase 11).

**Short names work only within the caller's own namespace search path.** resolv.conf lists `search <caller-ns>.svc.cluster.local svc.cluster.local cluster.local`, so a bare `traefik` resolves only from *inside* the `traefik` namespace; from another namespace (cloudflared lives in `cloudflare`) you need at least `traefik.traefik`, and cross-namespace config should always use the full FQDN so it can't be tripped up by the caller's search list.

**A ClusterIP is virtual — it lives on no interface; kube-proxy makes it work.** `traefik`'s ClusterIP `10.43.86.233` is assigned to no NIC. kube-proxy programs iptables/IPVS on every node so packets to `ClusterIP:port` are DNAT'd to a real backing **pod IP** (from the Service's EndpointSlice — here `10.42.2.6`), load-balanced across endpoints when there are several. That indirection is exactly why the Service name/IP stay stable while pods churn (pods are cattle; their `10.42.x` IPs change on every reschedule).

**Dial the Service `port`, never the container's `targetPort`.** The `traefik` Service exposes `port: 80` mapped to `targetPort: 8000` (the container's listen port). You connect to the *Service* port (80); kube-proxy translates 80→8000 en route to the pod. Connecting to `ClusterIP:8000` reaches nothing — there is no kube-proxy rule keyed on 8000, only on 80/443 (verified live: `:80`→404-from-Traefik, `:8000`→connection refused). DNS and port are independent coordinates: a name can resolve perfectly and the connection still fail on the wrong port. The Service→container link is often by *name* (`targetPort: web`), resolving to the container's named port rather than a hardcoded number.

**Why Traefik's container listens on 8000/8443, not 80/443.** The pod runs **non-root**, and binding to privileged ports (<1024, like 80/443) needs root or `CAP_NET_BIND_SERVICE`. So Traefik binds high, unprivileged ports (8000/8443) inside the container, and the Service maps the conventional 80/443 onto them. 8000 isn't a quirky choice — it's just Traefik's default unprivileged listen port, and the chart's default Service exposes it on 80 (this is the whole origin of the `:80`-vs-`:8000` confusion in Phase 11).

**A Service is a standing query, not a container of pods — so N Services fronting one pod is the default consequence, not a special arrangement.** There is no membership list inside a Service and no pod-side registration: pods carry *labels* (stamped once, by whoever created them — the Helm chart here), Services carry *selectors* (label predicates), and the **EndpointSlice controller** continuously materializes each query's current answer into `EndpointSlice` objects. Seen live: `traefik` (Helm's, ClusterIP) and `traefik-internal` (ours, Tailscale LoadBalancer) declare the *identical* selector, neither knows the other exists, and `kubectl get endpointslices -n traefik` shows two slices — one per Service — both resolving to the same pod IP. The pod ran unmodified and unrestarted while its Service count went from one to two. Accurate analogy: pods are rows in a table, each Service is a *view* — and rows don't know which views select them. The direction of the relationship is the point: the pod's owner doesn't control who selects it, which is exactly why a hand-written Service could front chart-managed pods without touching the chart (`k8s/traefik/tailscale-service.yaml`).

**Multiple Services over one pod can select different *ports* — different doors into the same process.** The two Traefik Services don't just share the pod: `traefik` maps 80/443 → the pod's 8000/8443, while `traefik-internal` exposes *only* 443 → 8443 (visible in each slice's port list). Since Traefik's container ports are its *entrypoints* (`web`/`websecure`), each Service effectively opens a different subset of doors — the public Service exposes the plain-HTTP door for cloudflared, the internal one exposes only the TLS door to the tailnet. Same pod, different doors, enforced at the Service layer (ADR-0029).

**Standard Service load balancing is L4: random, blind, once per *connection*, decided in the node's kernel.** Three precision points that complete the ClusterIP picture. (1) DNS does zero balancing — CoreDNS returns the one ClusterIP to every caller, always; all balancing happens later. (2) The DNAT happens in the *node's* kernel, not the pod's — kube-proxy programs iptables/IPVS rules into **every node** (which is why it runs on every node: each needs the full Service→endpoints map locally), and the rewrite fires as packets traverse the client's own node's stack; all pods on a node share that rule set. kube-proxy is only the rule *compiler* — it is never in the data path; the kernel is the data plane. (3) The backend choice is weighted-random at TCP connection setup, and conntrack pins every later packet of that connection to the chosen pod. Blind spots that follow: HTTP keep-alive glues many requests to one pod (worst case gRPC/HTTP-2 — one long multiplexed connection = zero balancing), nothing upstream can *choose* a backend (no stickiness, no canary weights, no retry-against-a-different-pod), and L4 can't see requests at all (no per-request health awareness).

**kube-proxy and Traefik are two consumers of the same EndpointSlices — one compiles them into kernel NAT (L4), the other into a userspace L7 balancer.** Traefik doesn't use separate discovery: it watches the exact EndpointSlices kube-proxy watches, but being a reverse proxy it terminates the client's connection and originates its own to the backends — moving the balancing decision from "once per connection, in the kernel" to "once per *request*, in Traefik." Making a choice requires dialing pod IPs **directly** (through the VIP, the kernel would take the choice away), which is the prerequisite for every L7 feature: per-request round-robin, cookie stickiness, retries against a different pod, per-backend health checks, weighted canary splits. It's deliberate and reversible — Traefik's `nativeLB` option opts back into the ClusterIP path. Evidence from this cluster: the Phase 13 curl loop returned alternating pod `Hostname:`s **per request** (impossible for the kernel path on one keep-alive connection), and every whoami body shows `RemoteAddr: <traefik pod IP>` — Traefik's own connection, direct to the pod (see [[Platform Concepts]] for the full hop-by-hop paths).

## NetworkPolicy

**NetworkPolicy can restrict *egress*, not just ingress — and egress is the important direction for a public-facing pod.** cloudflared is the one component reachable from the internet, so the risk is it being used as a *pivot*: by default any pod can reach every Service in every namespace via cluster DNS. `cloudflared-restrict-egress` clamps its egress to exactly DNS (53), Traefik (8000), and Cloudflare's edge (7844 UDP+TCP, 443) — so a compromised exposed app can't walk to a database or the Traefik dashboard through the tunnel pod (ADR-0017, `k8s/cloudflared/networkpolicy.yaml`).

**`namespaceSelector`/`podSelector` are `LabelSelector`s — labels go under `matchLabels`, not bare.** `namespaceSelector: {kubernetes.io/metadata.name: traefik}` fails with `strict decoding error: unknown field`; it must be `namespaceSelector: {matchLabels: {kubernetes.io/metadata.name: traefik}}`. The `kubernetes.io/metadata.name` label is auto-stamped on every namespace by Kubernetes (since 1.22), so it's the reliable way to select a namespace by name.

**An egress rule's port is matched *after* Service DNAT — so it's the pod's `targetPort`, not the Service port.** The rule allows TCP **8000** to the `traefik` namespace even though cloudflared *dials* the Service on **80**: the policy is enforced after kube-proxy has already translated `ClusterIP:80 → podIP:8000`, and a `namespaceSelector` inherently matches the destination *pod*. So the wire the policy inspects carries `podIP:8000`. Same two-coordinate lesson as above, from two vantage points: cloudflared dials 80, the netpol matches 8000.

**This works on plain Flannel — no Cilium needed.** k3s bundles a NetworkPolicy controller, so egress enforcement is real even though the CNI is Flannel and Cilium is deferred (ADR-0011, ADR-0017).

## IngressRoute — Traefik's native routing CRD

**An IngressRoute is what wires a public hostname to a pod — the final hop in the public path.** It lives under `traefik.io/v1alpha1` (a CRD-added group) and the Traefik controller watches for it. The `match: Host(\`whoami.neovara.uk\`)` field tells Traefik "when you receive a request with that exact `Host:` header on the `web` entrypoint, forward it to this Service." The `Host()` backtick syntax is Traefik's own DSL, not a Kubernetes standard — Traefik's controller parses it, not the API server.

**One entrypoint per port, and the Service target is a regular ClusterIP.** The `entryPoints: [web]` field routes this rule to the `web` entrypoint (container port 8000, reachable via `traefik` Service port 80 → kube-proxy DNAT → pod 8000). The `services:` blocks target a regular `ClusterIP` Service by name + port. Adding app #2 means: one new tunnel public hostname + one new `IngressRoute` — the `cloudflared` Deployment never changes, Traefik doesn't need a restart.

**Validated end-to-end in Phase 13:** `curl https://whoami.neovara.uk` → edge → tunnel → cloudflared → traefik:80 → Traefik matched `Host(\`whoami.neovara.uk\`)` → whoami Service → pod. The whoami body confirms every hop: `Cf-Connecting-Ip` (your router's real IP, from Cloudflare), `Host: whoami.neovara.uk` (the header Traefik matched), `X-Forwarded-Host` (Traefik forwarded), `Hostname:` (which specific pod served it — alternating across replicas).

**Two IngressRoutes → one app is a different mechanism than two Services → one pod: routes reference a Service *by name*, not pods by labels.** `whoami` and `whoami-internal` both end in `services: [{name: whoami, port: 80}]` — no selector anywhere. An IngressRoute isn't a network object at all; it's *configuration data for Traefik*, compiled by its CRD provider into rows of an in-memory routing table. Two rows referencing the same backend is as unremarkable as any routing table, and no conflict is possible because a request matches on the `(entrypoint, Host)` pair — the two whoami routes differ in *both* coordinates (`web` + `whoami.neovara.uk` vs `websecure` + `whoami.in.neovara.uk`), so each door only ever carries its own hostname (`k8s/example-app/ingressroute.yaml`, `ingressroute-internal.yaml`).

**Which object multiplies depends on which layer differs: Services multiply per exposure mechanism, IngressRoutes per routing rule — the app multiplies for neither.** Traefik needed multiple Services because the two paths differ in *how traffic reaches it* (cluster-internal ClusterIP for cloudflared vs a tailnet device via the operator) — an exposure concern, a Service-layer thing. whoami needed only one Service because by the time traffic is inside Traefik, the paths differ only in *routing rules* — so IngressRoutes multiplied instead, and whoami receives identical plain HTTP on port 80 through either door, blissfully unaware two front doors exist (ADR-0018, ADR-0029).

**Traefik doesn't forward through the ClusterIP — it reads the Service's EndpointSlices and dials pod IPs directly.** The referenced Service acts as an *address book*, not a network hop: Traefik does its own load-balancing across the endpoint IPs. The evidence was in every whoami response all along: `RemoteAddr: 10.42.2.6:...` — the whoami pod sees the *Traefik pod's* IP connecting straight to it, never a ClusterIP. Same "a Service is a query" idea taken one step further: some consumers just read the query's answer and go direct.

**TLSStore: why an IngressRoute's `tls.secretName` wasn't enough, and how one default cert covers every internal app.** Traefik only accepts a `tls.secretName` from the IngressRoute's *own* namespace — but app routes live in their app namespaces (whoami → `default`) while the cert-manager wildcard lives in `traefik` (`internal-wildcard-tls`). The escape hatch is Traefik's `TLSStore` CRD: exactly *one* store, which must be named `default`, cluster-wide, whose `defaultCertificate.secretName` becomes the cert Traefik serves whenever a route enables TLS without naming one. So any IngressRoute with an empty `tls: {}` gets the wildcard — and because `*.in.neovara.uk` covers every internal name, internal TLS is zero-config per app: new app = one IngressRoute with `tls: {}`, no cert plumbing, ever (`k8s/traefik/tlsstore.yaml`, `k8s/example-app/ingressroute-internal.yaml`, ADR-0029). Validated live: Traefik presented `CN=*.in.neovara.uk` (Let's Encrypt, verified without `-k`) for a route in the `default` namespace.

## Persistent storage: the PV / PVC / StorageClass object chain

**Storage is the same object model as everything else — records in etcd, reconciled by controllers — and the chain has a fixed shape: pod → PVC → PV → driver → bits on disk.** A **PersistentVolumeClaim** is a *request* record ("I want 1Gi, ReadWriteOnce, from class X") and a **PersistentVolume** is a *supply* record (one actual provisioned volume); Kubernetes **binds** a claim to a volume and the pod only ever names the claim (`persistentVolumeClaim.claimName`). The load-bearing consequence: **data lifecycle is coupled to exactly the PVC — fully decoupled from pods.** Delete a pod, wait two days, restart it on a different node: same `claimName` → same PVC → same PV → same data (proven live with the smoke test: `/data/proof` survived pod deletion and rescheduling). The right Docker analogy is a *named volume* (`docker volume create` + `-v mydata:/data`), not a bind mount — an object with its own identity whose "daemon" is the whole cluster. The bind-mount analogy belongs to `local-path`, which really is just a directory on one specific node.

**`reclaimPolicy` is where the coupling deliberately ends: deleting the PVC is the actual data deletion.** Both Longhorn classes here use `Delete` — removing the claim destroys the Longhorn volume and its replicas. So the operational reflex: `kubectl delete pod` is always safe for data; `kubectl delete pvc` is the `rm`. (The smoke-test cleanup deletes the PVC explicitly for exactly this reason.)

**Dynamic provisioning is what a StorageClass is for; static provisioning is the other path, and `longhorn-static` is its marker.** Dynamic: PVC names a class → the class's CSI provisioner *creates* the volume and its PV on demand — nobody hand-writes PVs. Static: the volume already exists (created in Longhorn's UI, or restored from backup) and gets a hand-made PV wrapper. Longhorn's manager creates the `longhorn-static` StorageClass itself at runtime (it's not in the Helm chart's rendered output) purely to stamp such PVs — note it carries **no `numberOfReplicas` parameter**, because it never provisions anything; it exists so the dynamic provisioner keeps its hands off. Never referenced in an app manifest.

**"Replica" means two unrelated things, and only one of them lives in an app's manifest.** A Deployment's `spec.replicas` = how many *pods* run (compute redundancy). Longhorn's `numberOfReplicas` = how many *copies of a volume's data* exist on different workers (data redundancy). Orthogonal axes: picture a Postgres with `spec.replicas: 1` (normal for a database) on a 1-data-copy volume — the worker holding the copy dies, Kubernetes happily reschedules the *pod*, but there's nothing to attach. Pod replicas never protect data; data replicas do. And a PVC **has no replica-count field at all** — `numberOfReplicas` is a StorageClass *parameter*, which forces the next concept:

**StorageClasses are policy tiers; per-app storage policy is expressed by which class name a PVC states.** This cluster's tiers: `longhorn` (1 data copy — dev) and `longhorn-replicated` (2 copies, one per worker — prod), identical except for that one parameter (`k8s/longhorn/values.yaml`, `k8s/longhorn/storageclass-replicated.yaml`, ADR-0030). Same driver underneath; the class is pure policy. This is also why neither is the cluster default (ADR-0031): a PVC saying `storageClassName: longhorn-replicated` is an explicit, reviewable statement of the app's durability requirement.

**`volumeBindingMode` answers one question: must volume placement wait for pod placement?** `local-path` uses `WaitForFirstConsumer` out of necessity — the "volume" is a directory on whichever node the pod lands on, so provisioning *cannot* happen before the scheduler picks that node (this is why a bare `local-path` PVC sits `Pending`, which looks broken and isn't). Longhorn's classes use `Immediate` — a Longhorn volume isn't pinned to the consumer's node (any worker can attach it over the network), so there's nothing to wait for; the smoke-test PVC bound instantly with no pod.

## CSI — how storage plugs into Kubernetes

**CSI (Container Storage Interface) is the plug socket that lets out-of-tree storage systems drive the PV/PVC machinery — Longhorn is this cluster's first real CSI driver.** The interface splits storage's job into three verbs at three moments: **provision** (a PVC appears → create a volume), **attach** (a pod is scheduled → make the volume a block device on *that node*), **mount** (put a filesystem on it and bind it into the pod). `local-path` needs none of this complexity because it skips two of the verbs — nothing to attach, nothing remote to provision.

**The CSI deployment pattern: Kubernetes ships the generic halves as sidecars, the vendor ships only the driver.** The four `csi-*` Deployments (`attacher`, `provisioner`, `resizer`, `snapshotter`) are stock upstream components — they watch the Kubernetes API for PVCs/attachments/resize requests and translate them into gRPC calls against the vendor's driver. The per-node `longhorn-csi-plugin` pod is `3/3` for the same reason: `longhorn-csi-plugin` (does the actual mounts), plus two stock sidecars — `node-driver-registrar` (announces the driver to that node's kubelet over a socket) and a liveness probe. Any CSI driver — EBS, Ceph, NFS — looks like this same kit of parts.

**Pod count ×N means two different things, and which one applies is readable from the workload kind.** The DaemonSet-shaped components (`longhorn-manager`, `longhorn-csi-plugin`, `instance-manager`, `engine-image-*`) run ×2 because there are 2 schedulable nodes — that's *topology*, not HA, not tunable; a third worker makes them ×3 automatically, and each node genuinely needs its own (the manager reconciles *that node's* disks; the csi-plugin mounts on *whichever node a pod lands*; the instance-manager hosts the engine/replica processes living on *that node*). The Deployment-shaped CSI sidecars default to ×3 and that *is* HA — leader-elected hot standbys, only one active, and they're control-path only (provisioning operations, never data). Set to ×1 here: worst case a new PVC waits a few seconds while the pod reschedules (`k8s/longhorn/values.yaml`). Also note none of Longhorn's pods run on the control-plane node — they don't tolerate its taint, which is *correct by default*: only workers run pods that attach volumes.

## Longhorn — distributed block storage

**Longhorn is the operator pattern yet again, applied to storage: a vocabulary of CRDs (`volumes.longhorn.io`, `replicas`, `engines`, `nodes`…) reconciled by `longhorn-manager`, a DaemonSet that is simultaneously the control loop and each node's storage agent.** Same shape as cert-manager and the Tailscale Operator — `kubectl get volumes.longhorn.io -n longhorn-system` shows the real state of every volume as plain Kubernetes objects. Its distinctive design choice sits one level down: **every volume gets its own dedicated engine** — a tiny storage controller process serving just that volume, hosted inside the `instance-manager` pod on whatever node the volume is attached to. A crashing storage controller therefore has a blast radius of exactly one volume. The engine *binary* ships as a container image distributed by the `engine-image-ei-<hash>` DaemonSet — a separate long-running component (not a one-time pull) because upgrades run multiple engine versions side by side, live-migrating volumes one at a time.

**The data path is iSCSI — a 20-year-old SAN protocol reused so compute and data placement decouple.** When a pod uses a Longhorn volume: pod → ext4 → a local block device → that device is an iSCSI target served by the volume's engine → the engine fans writes out over the pod network to every replica, synchronously. **Replicas are pinned to nodes** (sparse files under `/var/lib/longhorn/replicas/` on the dedicated data disk from ADR-0021); **the engine is not** — it materialises wherever the pod is and reaches the data over the network. So a pod rescheduling to the other worker keeps its data with zero migration: new engine on the new node, reads/writes travel node-to-node. The caveat that follows directly: with 1 data copy this only works *while the copy's node is alive* — rescheduling ≠ node failure, and node failure with `numberOfReplicas: 1` is data loss (the dev/prod tier split of ADR-0030). Seen live: `sudo iscsiadm -m session` on the attached worker shows the engine's iSCSI session; the kernel prerequisites are why Ansible now ensures `iscsid` is enabled and running on workers (`ansible/roles/k3s_agent/tasks/main.yml`, `--tags longhorn`). There's also a `dataLocality` class parameter (`best-effort` migrates a replica toward the attached node over time) — left `disabled` here, where both "nodes" share one physical NVMe and the network penalty is theoretical.

**Replica anti-affinity is hard by default: two copies can never share a worker — which also caps `numberOfReplicas` at the schedulable node count.** The chart default of 3 on this 2-worker cluster would leave every volume permanently degraded; 2 is the honest maximum (and per ADR-0021's standing caveat, still *simulated* durability until real hardware — both workers live on one physical disk, so the 2-copy tier protects against a worker VM dying, not the NVMe dying).

**RWX is faked with NFS on top of a block volume — block devices are inherently single-writer.** `ReadWriteMany` makes Longhorn interpose a **share-manager** pod (an NFS server) in front of a normal volume; every consuming node mounts it as an NFS client — hence `nfs-common` in the Ansible prerequisites, installed before any RWX volume exists. This also explains `longhorn-manager` reading `2/2`: its second container, `pre-pull-share-manager-image`, only pre-downloads the share-manager image so a future RWX attach doesn't stall on an image pull. Multi-container pods generally: `N/M` in `get pods` is containers-ready, not replicas.

**The Longhorn UI ships with zero authentication and can delete volumes and wipe disks — so it gets only the internal front door, never a public twin.** `longhorn.in.neovara.uk` via the same second-front-door pattern as whoami-internal: `websecure` entrypoint, Tailscale-only Service, default TLSStore wildcard (`k8s/longhorn/ingressroute.yaml`, ADR-0033). The IngressRoute lives in `longhorn-system` because Traefik only resolves same-namespace Services by default.

**A restart count on a freshly installed operator is often a startup race, and `kubectl logs -p` is how you prove it.** `longhorn-manager` showed 2 restarts from its first minutes: it also serves admission webhooks, and a manager can come up before the webhook endpoints it depends on are reachable, crash, and restart into working order. `RESTARTS` + currently `Running` + stable = read the *previous* container's logs (`kubectl logs <pod> -p`) to confirm what it died of before assuming a real problem — same lesson as ADR-0025's `--previous`, now a reflex.

## Monitoring: Prometheus, the Operator pattern applied to metrics, and the pull model

**Prometheus pulls; nothing pushes to it.** This is the opposite direction from everything else built so far on this platform (cloudflared holds an *outbound* connection; Alloy, below, *pushes* logs to Loki). Prometheus itself periodically reaches out to each target's `/metrics` HTTP endpoint on a timer and scrapes whatever numbers are sitting there — nothing "sends metrics to Prometheus." This is why a component that never gets scraped is silent rather than erroring: there's no push to fail.

**ServiceMonitor/PodMonitor are the IngressRoute of the metrics world — the same "stop hand-editing a config file, declare a CRD instead" move, one layer down.** `kube-prometheus-stack` doesn't install bare Prometheus; it installs the **Prometheus Operator**, which reads `Prometheus`/`Alertmanager`/`ServiceMonitor` CRDs and regenerates Prometheus's actual scrape config itself. A `ServiceMonitor` says "scrape any Service matching these labels, on this port, every N seconds" instead of a hand-maintained YAML list of targets — same shape as an IngressRoute replacing a raw Ingress object, same operator-pattern already seen with cert-manager and Tailscale (CRD = vocabulary, controller = behavior).

**The selector chain has four hops, not three — and the outermost one is easy to miss.** `ServiceMonitor.spec.selector` matches a **Service**'s labels (not a pod's); the Service's own `selector` is what already matches the pod. So the full chain is pod → Service → ServiceMonitor → **Prometheus's own selector over ServiceMonitors**. That fourth hop is a field on the `Prometheus` CRD itself (`spec.serviceMonitorSelector`) — "of all the ServiceMonitors that exist anywhere in the cluster, which ones do I even look at?" It's easy to assume this one is always empty/permissive and stop reasoning at hop three, which is exactly the mistake made and later corrected here (see below).

**An empty label selector matches everything, not nothing — but `kube-prometheus-stack`'s chart does *not* default to empty, contrary to an earlier assumption in this project.** The chart flag is `serviceMonitorSelectorNilUsesHelmValues` (plus matching flags for PodMonitor/Rule/Probe), and it defaults to `true` — which auto-generates `serviceMonitorSelector: {matchLabels: {release: <helm release name>}}`, *not* `{}`. Confirmed the hard way: Longhorn's and Traefik's own bundled `ServiceMonitor`s (issue #25) existed correctly, with correct labels, matching correctly-labeled Services, with real endpoints behind them — and were still completely invisible, not even appearing in Prometheus's generated scrape config, because they don't carry `release: kube-prometheus-stack` (a label that only make sense on kube-prometheus-stack's *own* bundled ServiceMonitors). Fixed by explicitly setting all four `*SelectorNilUsesHelmValues: false`, which is what actually produces `serviceMonitorSelector: {}` (confirmed via `helm template` and live, `kubectl get prometheus ... -o jsonpath='{.spec.serviceMonitorSelector}'` → `{}`) (ADR-0039). Lesson: "the chart's own values" is not the same claim as "what the chart actually does by default" — the two were conflated here once already, and only checking the *rendered* `Prometheus` object, not the values file's comments, caught it.

**With the selector actually empty, the practical consequence holds as originally intended:** this Prometheus install never needs to be touched to pick up a new scrape target anywhere in the cluster — any namespace, any app, including a future Kiroku deployment — as long as *that thing* has a ServiceMonitor pointing at it. The only per-app work is instrumenting the target and writing one small CRD next to it.

**A `ServiceMonitor` with zero backing endpoints is invisible to Prometheus, not "down."** Verified live: `kube-prometheus-stack-kube-scheduler`/`-kube-controller-manager`/`-kube-etcd` all have real ServiceMonitors (the chart creates them by default, assuming a kubeadm-shaped control plane), but `kubectl get endpoints -n kube-system <name>` shows an empty `.subsets` for all three — k3s runs these components inside its single server binary, not as separately-labeled pods with the addresses these chart-generated Services expect to select. Because there's nothing behind the Service, these three never even appear in `/api/v1/targets` at all — a structurally different failure mode from a real target that's merely unreachable (which *would* show up, flagged `health: down`). Tracked as follow-on work (GitHub issue #25), not a bug in the install.

## Stateless exporters, container storage, and what `df -h` actually shows

**Every container gets a writable filesystem whether anything is declared for it in the pod spec or not — it's a property of the container runtime, not Kubernetes.** Below the PVC/PV machinery entirely, `containerd` gives every container a **writable overlay layer** stacked on the image's read-only layers — purely because OCI containers need *somewhere* to write temp files and sockets. It's backed by the node's own local disk (on this cluster, the OS disk under `/var/lib/rancher/k3s/agent/containerd/...` — not the separate Longhorn data disk). Critically it's ephemeral in a stricter sense than expected: a Kubernetes "container restart" doesn't restart a process in place, it destroys the container and creates a fresh one, wiping this layer every time — which is exactly why anything meant to survive a restart has to be an explicit volume.

**"Stateless" doesn't mean "no filesystem" — it means "nothing on that filesystem is worth keeping," and the two example components in this stack sit on opposite sides of an even finer distinction.** Checked both rendered pod specs directly. `kube-state-metrics` has exactly one volume mount — a TLS secret for its own metrics-endpoint cert — nothing else; `df -h` inside it would show only its own tiny writable overlay plus the tmpfs-backed secret mount (Kubernetes Secrets are always projected as tmpfs/RAM, never real disk). `node-exporter` is different: it mounts the **host's own** `/proc`, `/sys`, and `/` in, read-only (`hostPath` volumes to `/host/proc`, `/host/sys`, `/host/root`). That's not node-exporter's storage — it's a **read-only measurement window** into the real node, which is the actual mechanism by which it can report genuine host disk/CPU/memory numbers as Prometheus metrics. Categorically different from a component that owns and writes its own state.

**No `ephemeral-storage` request/limit is set anywhere in this stack today, so every pod's writable layer draws from one shared, unbounded pool per node.** Same knob family as `resources.requests.cpu`/`memory`, just unused so far here. The backstop if it fills up isn't a per-pod quota error, it's the kubelet's own **disk-pressure eviction** — pods get evicted cluster-wide to reclaim node disk space. Not a problem worth fixing today (nothing in this stack writes runaway temp files), but the honest state of things.

## Umbrella charts bundle independent projects — inconsistent values shapes are the visible seam, not a design flaw to normalize

**`kube-prometheus-stack` doesn't implement Prometheus, Alertmanager, or Grafana — it glues together three genuinely separate upstream projects, and each chose its own values.yaml ergonomics independently.** Prometheus and Alertmanager aren't rendered by plain Helm templates at all: they're managed by the **Prometheus Operator**, which reads `Prometheus`/`Alertmanager` CRDs and creates the real StatefulSets itself. The values under `prometheus.prometheusSpec`/`alertmanager.alertmanagerSpec` get passed almost verbatim into those CRDs' `.spec` fields — so `storageSpec.volumeClaimTemplate.spec.{storageClassName,...}` is *literally* a raw `PersistentVolumeClaimSpec` object, unwrapped, because the Operator's design philosophy is "expose the real Kubernetes shape, don't invent a simplified one." Grafana, by contrast, is a *third*, unrelated chart (`grafana/grafana`) pulled in as a subchart — no CRD, no Operator, just a Deployment + PVC — whose own maintainers chose a flattened convenience API instead: `persistence.enabled`/`storageClassName`/`size`. Even the tiny `storageSpec` (Prometheus) vs `storage` (Alertmanager) naming mismatch isn't the chart authors' inconsistency — it's inherited straight from two CRDs designed independently, at different times, by the same upstream Operator project. Three teams, three opinions, one umbrella chart — the mismatch is the visible seam of that bundling, not something to fix.

**A chart's bundled dependency existing in `Chart.yaml` doesn't mean it gets installed — each has a `condition:` gate, and the actual default matters more than the dependency's presence.** Checked directly: `grafana/loki`'s `Chart.yaml` lists `minio` as a dependency, condition `minio.enabled` — but that key defaults to `false` (confirmed in the chart's own `values.yaml`), so nothing about MinIO ever renders unless that flag is explicitly flipped, regardless of it being "bundled." Separately, `grafana/loki` has **no Grafana dependency at all**, despite an initial assumption that it did — that bundling exists only in the older, legacy `loki-stack` chart, a different project entirely. Two lessons in one: check the *condition's default*, not just whether a dependency is listed; and don't assume a chart's shape from a similarly-named chart's shape.

**Chart defaults are not sized for your cluster — always check bundled subcomponents individually, not just the headline component.** The Loki chart's own resource sizing (`singleBinary`) was set deliberately small, but its two bundled Memcached-based read-path caches were left untouched — and `chunksCache` defaults to `allocatedMemory: 8192` (**8Gi**, clearly a production-scale default), which genuinely failed to schedule on this cluster (`0/3 nodes are available ... Insufficient memory` against two 9Gi-allocatable workers, confirmed live via `kubectl describe pod`). `resultsCache` defaults to 1Gi — smaller, so it silently fit and ran without ever raising an error, masking that it was equally unexamined. Neither cache is required for correctness (both are pure read-path speedups; Loki works without them, just re-reads chunks from its PVC on cache misses) — right-sized down to 256Mi each rather than disabled outright, matching how every other component in this stack was sized: lean, not zero.

## DaemonSet vs. Deployment — two placement models, and why splitting a workload N ways doesn't divide its cost by N

**A Deployment places N interchangeable pods wherever the scheduler finds room; a DaemonSet places exactly one pod per node, automatically, with no count to set.** Node joins → a DaemonSet pod appears there; node leaves → its pod goes. `node-exporter` is a DaemonSet for exactly this reason — reading `k3s-worker-1`'s own `/proc` requires a pod actually *on* `k3s-worker-1`. This is the pattern for anything that must be physically present per-node (a metrics exporter, a CNI agent, historically Promtail); a Deployment is the pattern for stateless, redundant, location-agnostic replicas.

**Pairing a DaemonSet (many collectors) with a whole-cluster query (each collector doing everyone else's job too) is the worst of both worlds — caught here before ever going live.** Alloy's initial config discovered *every* pod cluster-wide (`discovery.kubernetes "pods" { role = "pod" }`) while running as a 3-node DaemonSet — meaning each of the 3 pods independently found and shipped the *same* logs, tripling ingestion with nothing to deduplicate it. The DaemonSet shape itself wasn't the problem; pairing it with a cluster-wide (rather than per-node-scoped) query was. The two legitimate fixes are: scope each DaemonSet pod to only its own node's on-disk log files (the filesystem itself then partitions the work, no coordination needed), or drop to a single collector reading the whole cluster once via the API — chosen here, since a homelab's log volume doesn't justify the first option's extra moving parts (a taint toleration, a hostPath mount, a downward-API node-name filter).

**Splitting a workload across N nodes does not divide its resource cost by N when a fixed per-process baseline dominates — the actual criterion is whether per-node throughput would be big enough to matter.** Alloy's cost has two components: a roughly-constant per-pod baseline (the Go runtime, its own connection/watch to the Kubernetes API) plus a throughput-proportional part (bytes of logs actually shipped per second). At homelab log volumes the baseline dominates, so 3 per-node pods would each cost close to a *full* baseline — roughly **3x the aggregate** resource spend for the same total work, not a third each as naive division would suggest. A single Deployment replica was the cheaper shape here specifically because throughput is low; the moment a genuinely log-heavy workload arrives, that inequality flips and the DaemonSet-plus-file-tailing shape starts paying for itself. (ADR-0038.)

## Logs: Loki's design, and the API-vs-file collection tradeoff

**Loki only indexes labels, not the log text itself — the reason it's viable at homelab scale where a full-text index (ElasticSearch's model) wouldn't be.** Same label vocabulary Prometheus uses (namespace/pod/container). The actual log lines are stored as compressed, unindexed chunks — cheap to write, and search happens by first narrowing to a label set, then grepping within it, rather than a global inverted-text-index lookup on every word ever logged.

**Two ways to read a pod's logs without physically running on its node — Alloy chose the same one `kubectl logs` uses.** `loki.source.kubernetes` streams logs through the Kubernetes API (apiserver → kubelet), the identical mechanism `kubectl logs` uses — meaning one Alloy replica anywhere in the cluster can read *every* pod's logs, including ones on nodes it never runs on. This is what let a single Deployment (not a DaemonSet) cover the tainted, un-schedulable `k3s-server-1` for free: no toleration needed, because Alloy was never trying to run *there*, only to ask the API for what's already running there. Confirmed the RBAC actually supports this before relying on it: the chart's default ClusterRole already grants `get`/`list`/`watch` on both `pods` and `pods/log` — the specific permission this mechanism needs, verified by rendering the ClusterRole rather than assuming it.

## Reproducibility and verification: pinning names instead of discovering them, and a blind spot in `helm template`

**A Kubernetes Service's name from a Helm chart is not random — it's deterministic from a template, and pinning it removes an entire category of manual step.** The initial approach to wiring Alloy → Loki and the Grafana IngressRoute → Grafana was: install, `kubectl get svc` to discover whatever name the chart's default `<release>-<chart>` template produced, then hand-edit the next file. Real charts expose `nameOverride`/`fullnameOverride` specifically to avoid this — set before install, not discovered after. Applied here: `fullnameOverride: grafana` and `fullnameOverride: loki` (confirmed by rendering each chart locally and grepping the resulting Service's `metadata.name` before ever touching the cluster) make every downstream cross-reference a static string. Prometheus's Service name was left un-pinned on purpose — the *only* override available for it is the whole-release `fullnameOverride`, which (confirmed via `--set fullnameOverride=...`) renames every other object in the release too, for no benefit — but it's equally reproducible, since it's derived from the release name (`kube-prometheus-stack`) that the documented install command already fixes. (ADR-0035.)

**`helm template` validates Helm/YAML structure — it has no idea what's inside a string value, which is exactly the blind spot that let a real bug through.** Every monitoring manifest was dry-run rendered with `helm template` before ever touching the cluster, and all of them came back clean — including Alloy's `alloy.configMap.content` field, which is a multi-line *string* containing an entirely separate configuration language (Alloy's own "River" syntax). `helm template` only checks that the string is valid YAML content inside a ConfigMap; it can't parse River, so a YAML-style `# comment` inside that string (River's actual comment syntax is `//`, Go/C-style) rendered as perfectly valid YAML and only crashed once Alloy's own parser tried to read it at pod startup (`Error: ... illegal character U+0023 '#'`, caught via `kubectl logs`, not the earlier dry-run). General lesson: a templating dry-run only validates the *outer* language, never a second language embedded inside a string value — River here, but the same gap would apply to embedded shell scripts, SQL, or any other DSL passed through as chart values.

## Access control: a deliberate absence vs. a deliberate removal

**Two dashboards with "no login" on this platform arrived at that state by opposite paths, and the difference matters for the risk being accepted.** The Longhorn UI (ADR-0033) ships with *zero* authentication built in — there was nothing to keep, so the decision was to accept network reachability (the Tailscale-only IngressRoute) as the only gate, because adding real auth would have meant bolting on new machinery. Grafana is different: it already had a working login for free (an auto-generated admin password, retrievable from a Secret) — turning it off (`auth.anonymous.enabled: true`, `org_role: Admin`, `auth.disable_login_form: true`) is an *active downgrade* of something that cost nothing to keep, traded for the convenience of skipping a login screen on a trusted tailnet. Same underlying "reachability is the authentication" model, but one accepted an unavoidable gap and the other removed a working safeguard — worth keeping straight before applying the same pattern to a third dashboard, since the blast radius differs too (Longhorn: destructive volume/disk operations; Grafana: read access to whatever ends up in application logs via the Loki datasource, plus dashboard/alerting write access under the granted role). (ADR-0036.)

## `Endpoints` — the object a Service actually resolves to, and what happens when nothing is a pod

**A Service's `selector` doesn't route anything by itself — it's the input to a separate object, `Endpoints`, that the endpoint-controller writes and continuously keeps in sync.** Every Service with a `spec.selector` gets a same-named `Endpoints` object, auto-managed: the endpoint-controller watches for pods matching that selector and writes their `podIP:targetPort` pairs into `subsets`. `kube-proxy` (and, for scraping, Prometheus) reads *that* object, never the Service's selector directly. This is one layer further down than anything touched before this pass — Services had always been treated as if they resolved directly to pods.

**k3s's control-plane components broke this assumption completely: kube-scheduler, kube-controller-manager, and etcd are threads inside the single k3s binary, not pods — so no pod ever matches, and the auto-managed `Endpoints` stayed permanently empty (`subsets: []`).** This is the same "structurally invisible, not down" case already documented above for these three components — the missing piece was *why* it's permanent: there is no pod anywhere in the cluster carrying the labels these chart-generated Services select on, because kubeadm-shaped assumptions (control-plane components as discrete, labeled pods) don't hold on k3s.

**The escape hatch: hand-author the `Endpoints` object instead of waiting for the controller to populate one.** Setting `kubeControllerManager.endpoints: [192.168.1.21]` (and the scheduler/etcd equivalents) in the Helm values makes the chart itself write a static `Endpoints` object — `subsets: [{addresses: [{ip: 192.168.1.21}], ports: [{name: http-metrics, port: 10257}]}]` — bypassing pod-selector discovery entirely and pointing straight at the node's real IP and the now-opened metrics port (`kube-controller-manager-arg: bind-address=0.0.0.0`, set via k3s's own server config, ADR-0040). Confirmed live: before this, `kubectl get endpoints -n kube-system kube-prometheus-stack-kube-controller-manager` showed `<none>`; after, it showed `192.168.1.21:10257`, and the corresponding Prometheus target flipped from nonexistent to `up`.

**The one real gotcha: an object can be "correct" and still fail to apply, if two different owners both think they manage it.** The auto-managed empty `Endpoints` objects already existed (Kubernetes always creates one for any selector-bearing Service, even with zero matches) — but the endpoint-controller isn't Helm, so they carried no `meta.helm.sh/release-name`/`release-namespace` annotations. When the chart tried to start explicitly managing an object with that exact name, `helm upgrade` refused: `exists and cannot be imported into the current release: invalid ownership metadata`. This is Helm's adoption safety check — it will not silently take over an object some other actor created, even one that's functionally empty and harmless to overwrite. Fixed by manually adding the two required annotations (`kubectl annotate endpoints ... meta.helm.sh/release-name=... meta.helm.sh/release-namespace=... --overwrite`), which retroactively tells Helm "you already own this," after which the upgrade proceeded normally. Deleting the object first was tried and didn't help — the endpoint-controller recreates an empty one almost instantly for any Service with a selector, faster than the next `helm upgrade` could run; adoption via annotation, not delete-and-race, is the actual fix.

## Grafana dashboards as labeled `ConfigMap`s — the same watch-by-label idea, without a CRD

**A `ConfigMap` is the plainest built-in Kubernetes object there is — a namespaced bag of key/value data — and this pass used it to reimplement the same "watch for a label, react" shape as a CRD-based operator, with no CRD at all.** kube-prometheus-stack's Grafana pod runs a sidecar container (`grafana-sc-dashboard`) whose entire job is polling the Kubernetes API — across *every* namespace, by chart default (`sidecar.dashboards.searchNamespace: ALL`) — for `ConfigMap`s carrying the label `grafana_dashboard: "1"`. When it finds one, it writes that ConfigMap's data as a JSON file into a shared directory and calls Grafana's own provisioning-reload API. No `kind: GrafanaDashboard` CRD exists; the "vocabulary" here is just a label convention, and the "controller" is a plain sidecar container polling on a timer, not anything registered with the API server as an extension. Same conceptual move as `ServiceMonitor`/`IngressRoute` (declare intent via a label/object, let something else reconcile it into the real system) — just implemented one rung down the sophistication ladder, with a built-in kind instead of a custom one.

**A dashboard's own embedded datasource-variable syntax is a second declarative-config layer, with its own separate blind spot — orthogonal to whether Grafana notices the ConfigMap at all.** Grafana dashboard JSON (as exported from grafana.com) commonly references its datasource via a template variable like `${DS_PROMETHEUS}`, meant to be filled in by whatever imports it. The *UI* import flow prompts you to pick a datasource and substitutes the token. File-based provisioning (the ConfigMap-and-sidecar route above) does **not** do this substitution — it just hands Grafana the raw JSON. Traefik's official dashboard (grafana.com ID 17347) happened to work anyway, because its `templating.list` already carried a `type: datasource` variable with `current.value` pre-filled to the literal string `"Prometheus"` — coincidentally the exact name this chart gives its own provisioned datasource, confirmed by reading the rendered datasource ConfigMap first. Longhorn's dashboard (ID 17626, an older schema) had no such pre-filled variable — just 30 bare `"datasource": "${DS_PROMETHEUS-LONGHORN}"` string references with nothing to resolve them — so those needed a direct find-and-replace to the literal string `"Prometheus"` before the ConfigMap would render a working dashboard rather than 30 panels erroring "datasource not found." Same category of lesson as the River `#`-vs-`//` bug (ADR-0038's era): a file that is *itself* a second declarative language can carry its own unresolved placeholders that no amount of validating the outer YAML/ConfigMap will ever catch — only inspecting the specific mechanism (here: is there a template variable with a default, or not) tells you which dashboards need edits and which don't (ADR-0041).
